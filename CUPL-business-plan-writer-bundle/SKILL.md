# CUPL 创业计划书撰写 Skill（自包含捆绑包）

端到端引导用户完成一份专业、学术规范的创业计划书。
**本捆绑包自包含所有功能——不需要安装任何外部技能。**
所有步骤必须严格执行，不可跳过。

---

|-----------|-------------|

---

## 执行总览

```
Step 0: 加载规范 → 读取要求文件或切换全球模板
Step 1: 项目定位 → 内建脑暴诊断 + 结构化引导
Step 1.5: 设计风格 → StyleKit 120+风格速查 + 五轮推荐
Step 2: 产品打磨 → 内建全套模板（TAM/SAM/SOM、BMC、SWOT、PRD等）
Step 3: 技术方案 → 三级分流 + 五阶段内建工程流程 + 安全门禁
Step 4: 数据调研 → 强制 WebSearch 权威数据，记录 URL
Step 5: 撰写输出 → 逐章撰写，GB/T 7714 引用，内建 HTML 模板生成
Step 5.5: 去 AI 痕迹 → 12 条中文专项规则，逐段评分
Step 6: 质量检查 → 答题板核查 + 深度审计 + 评分量规
Step 7: 路演答辩 → PPT 大纲 + 答辩陈词 + 预演问答
```

每完成一步，向用户汇报进展，确认后进入下一步。

**严禁跳过 Step 0-2 直接写计划书。**

## Step 0: 加载规范文件

### 0.1 检测文件

搜索当前项目根目录下是否存在以下文件：

```
**/创业计划书规范要求.docx
**/创业计划书规范要求.*
**/business-plan-requirements.*
**/创业计划书要求.*
```

### 0.2 加载规则

**情况 A — 找到规范文件：**

1. 读取并解析规范内容，提取：必须包含的章节列表、字数/页数要求、评分标准、答题板/必答题清单、格式要求、提交格式
2. `.docx` 文件：`npx mammoth "文件路径"` 提取文本
3. `.doc` 文件：使用 Node.js UTF-16LE 字节扫描提取文本
4. `.pdf` 文件：直接 Read 或 `npx pdf-parse`
5. 向用户展示提取的规范摘要（含答题板清单），确认无误后继续

**情况 B — 未找到规范文件：**

告知用户将采用经过验证的创业计划书标准撰写（Stanford GSB / Y Combinator / HBS 标准）。

**比赛型 BP vs 真创业 BP 的本质区别：**
- 比赛型 BP：框架完整、面面俱到——读完记不住核心论点
- 真创业 BP：一个核心洞察贯穿全文、对单位经济模型有偏执般的细节、诚实面对风险

**三条原则：**
1. 每段话必须为你的核心论点添砖加瓦，否则删掉
2. 每个数据必须解释它意味着什么机会或风险，不堆数字
3. 诚实 > 完美

标准 13 章模板：封面 → 目录 → 执行摘要 → 企业介绍 → 产品与服务 → 行业与市场分析 → 营销计划 → 管理团队 → 运营计划 → 财务计划 → 风险分析 → 退出策略 → 附录

## Step 1: 项目定位

### 1.0 头脑风暴

在对用户做结构化问答之前，先用开放式对话帮用户理清思路。

**脑暴流程：** 读取 `references/built-in-toolkit.md` §一，执行 YC 创业诊断六问。逐问提问，得到回答后再下一问。

六问结束后总结沉淀：
- 用户真正想解决的问题是什么
- 目前的解决方案是什么
- 用户对这个项目的野心有多大

向用户确认总结，然后进入 1.1。

### 1.1 发现用户意图

询问用户当前情况：
- A) 我有明确的创业项目方向 → 跳转 1.2
- B) 我有几个方向但还不确定 → 跳转 1.3
- C) 我完全没有头绪 → 跳转 1.4

### 1.2 用户有明确方向

4 轮结构化引导（产品形态 → 目标用户与痛点 → 团队优势 → 当前阶段），每轮提供选项 + 自定义补充。完成后复述对项目的理解，确认无误进入 Step 1.5。

### 1.3 用户有几个方向

用户列出 2-5 个方向，按内建工具箱 §一"五维度方向扫描"逐一评估（市场容量/竞争格局/技术壁垒/政策环境/团队匹配），推荐 1-2 个最有潜力的方向。

### 1.4 用户完全没想法

用 WebSearch 搜索当前最具潜力的创业赛道，筛选 3-5 个有数据支撑的方向建议（含市场规模+URL、为什么现在适合、大学生切入点）。用户选择后回到 1.2。

---

## Step 1.5: 设计风格选择

### 风格推荐流程

1. 读取 `references/stylekit-styles.md` 获取 120+ 风格速查表
2. 根据 Step 1 收集的项目信息，提炼 3 个品牌气质关键词
3. 五轮推荐制：每轮展示 3 个风格方案（每轮来自不同类别），共最多 15 个方案
4. 用户选定后，如可访问网络，用 WebFetch 读取 `https://stylekit.top/api/styles/{slug}` 获取完整 Token
5. 网络不可达时，使用 `references/stylekit-styles.md` 中的颜色/字体/阴影/圆角参考
6. StyleKit 提供设计 Token，内建工具箱 §三"品牌深化规则"做品质提升

---

## Step 2: 产品打磨

### 2.0 品牌体系设计

用问答引导用户明确：产品命名与故事、品牌 Slogan、商标注册情况、品牌调性关键词、VI 概要。使用内建工具箱 §二"品牌策略模板"。

### 2.1 功能深度挖掘

6 轮结构化问答（功能补充 → 优先级排序 → 盈利模式 → 运营服务 → 增长策略 → 国际化扩展），每轮一个主题，不跳不混。

### 2.2 产品分析执行

使用内建工具箱 §二中的全套模板，按以下顺序逐项完成：

1. **TAM/SAM/SOM 计算** — 使用计算模板，标注数据来源 URL
2. **PESTEL 分析** — 使用 PESTEL 模板
3. **用户画像** — 使用 Persona 模板，产出 2-3 个
4. **竞争定位** — 使用竞争分析表，至少 3 个竞品
5. **商业模式画布** — 使用 BMC 9 要素模板
6. **SWOT 分析** — 每维至少 3 个要点
7. **品牌策略** — 使用品牌策略模板
8. **PRD 文档**（技术型项目）— 使用 PRD 模板

每项完成后向用户展示摘要。所有数据来源标注 URL。

### 2.3 数据质量要求

- 所有市场数据来自最近 3 年
- 来源优先级：政府统计 > 行业报告 > 咨询公司 > 媒体报道
- 禁止编造数据。搜不到精确数据时使用区间估算并标注"估算依据"

## Step 3: 技术方案

### 3.1 适用性判断（三级分流）

- **路径 A（完整技术方案）**：软件/SaaS/App/平台类、硬件/智能设备类 → 执行 3.2-3.7
- **路径 B（轻量级技术方案）**：混合型项目（服务+小程序/App）→ 仅做 3.2-B 轻量拆解 + 3.7 内置流程
- **路径 C（跳过）**：纯服务型、纯内容/IP 型 → 1-2 段简述即可

### 3.2 PRD 需求拆解

使用内建工具箱 §二"PRD 模板"产出 PRD，然后按模块拆解（前端/后端/AI/基础设施），每模块标注依赖和优先级。

**路径 B 在此停止**，产出简化版：技术栈（一句话）+ 架构草图（Mermaid）+ 开发路线图（3 个里程碑），进入 Step 4。

**路径 A 继续：**

产出 2-3 套实现方案（极简验证/标准工程/极致架构），含技术栈、开发周期、团队需求、成本估算、风险。用户选择后进入 3.3。

### 3.3-3.4 五阶段执行 + 安全门禁

读取 `references/built-in-toolkit.md` §三，按以下五阶段执行：

| 阶段 | 使用工具 | 关键产出 |
|------|---------|---------|
| 一：技术规划 | 选型矩阵 + ADR 模板 + Grill-Me 5 问 | 技术选型、ADR(≥3条)、风险点、规划移交摘要 |
| 二：架构设计 | 架构评审清单(10项) + ADR 模板 | 架构图(Mermaid)、API设计、数据模型、架构移交摘要 |
| 三a：原型验证 | 原型构建指南（可丢弃） | 可运行原型，仅验证可行性 |
| 🔒 安全门禁 | 安全清单（6类检查） | 全部通过方可进入三b |
| 三b：正式实现 | TDD 流程 + HTML 模板 | 测试套件、生产级代码、实现移交摘要 |
| 四：评审验证 | 代码审查清单 + 诊断流程 + 验证清单 + 成本回溯 | 评审报告、修复记录、成本回溯结论 |

**阶段移交摘要格式：** 每阶段结束写 3-5 行：关键决策、产出位置、留给下阶段的风险。

### 3.5 降级策略

所有流程均使用内建工具箱，不存在外部依赖缺失问题。如果用户 Agent 不支持某种工具（如无法执行 Bash 命令来生成 Mermaid），告知用户手动完成该步骤。

### 3.6 安全门禁

如涉及认证/支付/用户数据/第三方 API/UGC，使用内建工具箱 §三的安全清单逐项检查。高危项必须修复才能进入三b。

### 3.7 全内置后备流程

当用户 Agent 功能受限时（如无法执行代码），使用内建工具箱 §三的纯文档流程：架构图(Mermaid) → 技术选型 → ADR(≥3条) → API/数据模型 → 开发路线图 → 风险分析(≥5项) → 安全自查 → 资源估算。

## Step 4: 数据调研

### 4.0 WebSearch 可用性强制验证

MUST 执行：搜索当前年份相关的行业关键词，确认返回了含 URL 的真实搜索结果。WebSearch 不可用则暂停，询问用户是否接受使用训练知识替代。

### 4.1 六大维度搜索

对以下维度逐一搜索真实数据：

| 维度 | 搜索模板 | 必须记录 |
|------|---------|---------|
| 行业规模 | "XX行业 市场规模 2025 2026" | 具体数字 + CAGR + URL |
| 政策环境 | "XX产业 国家政策 2025 2026" | 政策文件名 + 文号 + URL |
| 消费趋势 | "XX消费趋势 2025 2026 报告" | 趋势数据 + 报告来源 + URL |
| 竞争格局 | "XX行业 市场份额 2025" | CR5 + 竞品详情 + URL |
| 技术发展 | "XX技术 发展趋势 2025 2026" | 技术节点 + 成熟度 + URL |
| 投融资 | "XX赛道 融资 2025 2026" | 融资金额 + 机构 + URL |

### 4.2 竞品深潜

对 2-3 个核心竞品搜索：融资历史、团队背景、产品数据、商业模式、用户评价。

### 4.3 财务对标

找 2-3 家可比公司的实际经营数据：收入规模、成本结构、单位经济模型。

### 4.4 学术文献

搜索 5-8 篇相关学术文献，记录完整题录（用于 GB/T 7714 引用）。

### 4.5 数据充分性闸门

7 项检查全通过才进入 Step 5。使用内建工具箱 §五的数据记录格式。缺什么补什么，不急着写。

## Step 5: 撰写与输出

### 5.0 写作铁律

1. 成文成段，严禁分点列举（计划书不是提纲）
2. 15,000-25,000 字规模
3. 数据必须配分析，分析必须出结论，结论必须连回商业逻辑
4. 图表为辅，不能替代文字
5. 先写后改，全部写完后用 Step 5.5 去 AI 痕迹

### 5.1 章节结构

标准 13 章：封面 → 目录 → 执行摘要 → 企业介绍 → 产品与服务 → 行业与市场分析 → 营销计划 → 管理团队 → 运营计划 → 财务计划 → 风险分析 → 退出策略 → 附录。如规范文件有不同要求，以规范文件为准。

### 5.2 引用规范

默认 GB/T 7714 顺序编码制。若检测到法大格式规范文件，改用脚注制（详见 `references/cupl-thesis-format.md`）。

### 5.3 逐章撰写

每章写完向用户展示，确认后继续。执行摘要最后写。

### 5.4 HTML 生成

使用内建工具箱 §三"HTML 内建模板"作为基础骨架，将 Step 1.5 选定的 StyleKit Token 映射为 CSS 变量。

**必须包含：** 封面、锚点目录、页眉页脚、正文排版、编号图表、规范引用、参考文献列表、打印友好 CSS（@media print — A4）、响应式设计。

**法大格式：** 检测到法大规范时，CSS 按 `references/cupl-thesis-format.md` 生成（三级汉字标题、脚注、页眉"中 国 政 法 大 学 本 科 生 毕 业 论 文"）。

HTML 保存为 `创业计划书.html`。

### 5.5 中文去 AI 痕迹化

先写后改。全部章节写完后回头做去痕迹处理。

**Step A: 执行 12 条规则**

逐一执行以下规则（详见内建工具箱 §四）：

1. 破折号严格限制（全文 ≤ 5 处）
2. 双引号限制（除直接引用外不使用）
3. 禁用 AI 标志词（"值得注意的是"、"综上所述"、"不仅…而且…"等）
4. 句式多样性（连续 3 句不以相同结构开头）
5. 具体优于抽象（用数字替代模糊形容词）
6. 删除水词（"全方位"、"多层次"、"赋能"等）
7. 专业术语自然化（先解释再给术语）
8. 中文标点规范（全角标点、中文引号）
9. 段落结构多样化（不只"总-分-总"）
10. 段落长度有起伏（1-2句短段 + 8-10句长段）
11. 语调分章节校准（执行摘要自信、行业分析冷静、财务保守、风险坦诚）
12. 朗读测试

**Step B: 去痕迹质量检查**

随机抽取 5 段正文（每段 200 字以上），逐项评分。不通过则重新修订。

**Step C: 用户确认**

将去痕迹后的关键段落展示给用户确认："这些文字读起来自然吗？还有 AI 味吗？"

## Step 6: 质量检查

### 6.0 答题板逐项核查

如果规范文件有答题板，逐项核对，100% 覆盖。

### 6.1 格式与完整性检查

逐项检查：所有章节包含、顺序正确、字数达标(15,000-25,000)、数据有 URL、引用规范、参考文献 5-8 篇、封面信息齐全、目录锚定、HTML 打印正常、语法错别字、图表编号连续。

### 6.2 深度审计

逐章深度检查（执行摘要可复述性、行业分析数据→分析→结论链、财务有对标+单位经济模型、风险具体可操作）。

### 6.3 逻辑流检查

从执行摘要到附录，检查全书逻辑链条是否连贯。

### 6.4 真创业 BP 对标

用核心洞察检验、行业理解检验、诚实度检验、论点聚焦检验。

### 6.5 评分量规

使用内建工具箱 §六的评分表自评（8 维 × 10 分）。总分 < 55 → 不提交，回头改。55-65 → 可提交。> 65 → 有说服力的计划书。

### 6.6 完成报告

全部通过后输出完成报告（文件位置、总字数、章节数、数据引用数、学术文献数、深度评分）。

---

## Step 7: 路演与答辩准备

### 7.1 路演 PPT 大纲

按商业路演标准生成 12 页 PPT 大纲（痛点→方案→产品→市场→商业模式→竞争优势→增长→壁垒→团队→财务→结束页）。每页不超过 3 个要点，字号 ≥ 24pt。

### 7.2 答辩陈词

生成逐字稿（1000-1200 字），格式为"讲什么 + 翻到第几页"。开头 30 秒直接进入痛点画面，不寒暄。

### 7.3 常见问题预演

预判 15-20 个最可能被问的问题（投资人视角 + 老师/评委视角），每问准备 150-200 字回复。结构：一句话结论 + 一个数据/事实 + 一个"所以"。

### 7.4 路演后材料

执行摘要（1 页 PDF）+ 联系方式 + 下一步行动引导。
