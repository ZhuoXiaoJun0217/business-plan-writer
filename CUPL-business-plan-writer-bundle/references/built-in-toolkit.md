## 一、头脑风暴与产品诊断

### YC 创业诊断六问（替代 office-hours Startup Mode）

依次向用户提问，每问得到回答后再下一问：

1. **问题诊断**：你最后一次亲身遇到这个问题是什么时候？当时你是怎么解决的？花了多少时间/钱？
2. **用户画像**：谁最痛恨这个问题？他们现在在用什么东西拼凑着解决？（哪怕是用 Excel + 微信群）
3. **市场规模**：粗略估算一下，有多少人和你遇到同样的问题？他们每个人愿意花多少钱解决？
4. **竞争格局**：你觉得目前解决这个问题最差的方案是什么？它差在哪？
5. **技术/运营护城河**：如果有人看到你做的，三个月内能不能抄出来？如果能，你怎么办？
6. **野心程度**：如果这个项目完全成功，你希望它变成什么样？如果失败，最可能的原因是什么？

六问结束后总结沉淀，确认用户的核心洞察。

### 脑暴三问（降级版）

1. "你上一次亲身遇到这个问题是什么时候？当时你是怎么解决的？花了多少时间/钱？"
2. "如果有人已经在做这件事了，你觉得他们做得最差的地方是什么？"
3. "如果这个项目失败了，最可能的原因是什么？"

### 五维度方向扫描

当用户有几个方向时，对每个方向做快速扫描：

| 维度 | 评估内容 | 评分 (1-5) |
|------|---------|-----------|
| 市场容量 | 粗略 TAM 量级（亿/十亿/百亿） | |
| 竞争格局 | 红海/蓝海/新兴 | |
| 技术壁垒 | 高/中/低 | |
| 政策环境 | 鼓励/中性/限制 | |
| 团队匹配 | 强/中/弱 | |

### TAM/SAM/SOM 计算模板

```
TAM（Total Addressable Market — 总可用市场）：
  计算逻辑：[用户总数 × 年均消费]
  = [X 亿人] × [Y 元/年]
  = Z 亿元

SAM（Serviceable Addressable Market — 可服务市场）：
  计算逻辑：[TAM × 目标细分占比 × 地域可达率]
  = Z 亿 × [a%] × [b%]
  = W 亿元

SOM（Serviceable Obtainable Market — 可获得市场）：
  计算逻辑：[SAM × 渗透率预估（基于竞品对标）]
  = W 亿 × [c%]
  = V 亿元

数据来源：
  - 用户总量：[来源名称, URL]
  - 年均消费：[来源名称, URL]
  - 渗透率对标：[竞品名称, 数据来源, URL]
```

### 架构评审清单（替代 /plan-eng-review — 10项检查）

```
□ 1. 数据流是否闭环？（用户→前端→API→数据库→返回，每步清晰）
□ 2. 状态管理方案是否合理？（全局状态 vs 局部状态的边界在哪）
□ 3. 错误处理是否完整？（网络失败、超时、服务端500、数据为空）
□ 4. 认证授权是否清晰？（谁可以访问什么）
□ 5. API 设计是否 RESTful？（资源命名、状态码、分页）
□ 6. 数据库 schema 是否规范化？（至少 3NF）
□ 7. 是否有 N+1 查询风险？（关联查询是否用 join 或 batch load）
□ 8. 缓存策略是否明确？（什么缓存、缓存多久、什么时候失效）
□ 9. 日志和监控是否覆盖？（关键路径有日志、有报警阈值）
□ 10. 是否有单点故障？（关键服务是否有备份或降级方案）
```

### TDD 内建流程（替代 mattpocock tdd + superpowers TDD）

```
Red-Green-Refactor 循环：

1. RED：写一个会失败的测试（只测一个行为）
2. GREEN：写最少代码让测试通过（不管多丑）
3. REFACTOR：清理代码，消除重复，改善命名
4. 重复，每个循环 5-10 分钟

测试结构（AAA 模式）：
  Arrange → 准备数据
  Act → 执行操作
  Assert → 验证结果

集成测试清单：
  □ API 端点返回正确状态码
  □ 数据库读写正常
  □ 认证流程（注册→登录→访问受保护资源）
  □ 核心业务流程端到端
```

### 原型构建指南（替代 mattpocock prototype）

```
可丢弃原型 — 仅验证可行性，不追求代码质量：

1. 确定要验证的假设（最多 2 个）
2. 用最简单的技术栈（HTML/JS 单文件 或 Node.js 单文件）
3. 只实现核心流程的 happy path
4. 不写测试、不做错误处理、不考虑性能
5. 验证完后记录结论：假设成立/推翻/不确定
6. 扔掉代码，正式开发从头开始
```

### Grill-Me 内置问答（替代 mattpocock grill-me — 5轮追问）

```
对用户的每个技术决策，依次追问：
1. "如果这个选型被证明是错的，最可能是什么原因？"
2. "有没有更简单的方案能做到同样的事？"
3. "假如用户量是现在的 100 倍，瓶颈会出现在哪？"
4. "安全方面你最担心什么？"
5. "如果团队里最懂这个技术的人走了，怎么办？"
```

### Diagnose 诊断流程（替代 mattpocock diagnose）

```
1. Reproduce（复现）：能否稳定复现？记录精确步骤
2. Minimise（缩小）：最小化可复现的代码/场景
3. Hypotheise（假设）：提出 2-3 个最可能的原因
4. Instrument（检测）：加日志/断点/监控验证假设
5. Fix（修复）：针对确认的原因修复
6. Verify（验证）：确认修复后问题不再出现 + 没有引入新问题
```

### 品牌深化规则（替代 ui-ux-pro-max）

```
视觉品质提升清单：

□ 色彩一致性：全站使用 StyleKit Token，不超过 3 种主色
□ 字体层级：标题/正文/辅助文字的字体、大小、粗细形成清晰层级
□ 间距系统：使用 4px 或 8px 倍数制，不出现随意间距
□ 阴影层次：近处阴影淡、远处阴影深，光源统一
□ 圆角一致性：同类元素使用相同圆角
□ 动画克制：过渡动画 200-300ms，缓动函数一致
□ 空白利用：不要填满每一寸空间，呼吸感比信息密度重要
□ 响应式：手机/平板/桌面三端视觉一致
□ 打印友好：打印时不显示导航、按钮、动画
```

## 四、12 条中文去 AI 痕迹规则

见主 SKILL.md Step 5.5——这些规则已完全内联，无需调用外部 humanizer-zh。

执行顺序：
1. 破折号限制（全文 ≤ 5 处）
2. 双引号限制
3. AI 标志词过滤（"值得注意的是"、"综上所述"等）
4. 句式多样性（连续 3 句不重复开头）
5. 具体优于抽象（数字替代模糊形容词）
6. 水词删除（"全方位"、"多层次"等）
7. 术语自然化（先解释再给术语）
8. 中文标点规范
9. 段落结构多样化
10. 段落长度起伏
11. 语调分章节校准
12. 朗读测试

## 六、质量评分量规

| 维度 | 10 分标准 | 及格线 |
|------|----------|--------|
| 核心论点清晰度 | 一句话能说清，所有章节都服务于这个论点 | 7 |
| 数据质量 | 每条数据有来源 URL，有分析有结论，财务有对标 | 8 |
| 分析深度 | 不堆数据，每个数据解释了"然后呢？" | 6 |
