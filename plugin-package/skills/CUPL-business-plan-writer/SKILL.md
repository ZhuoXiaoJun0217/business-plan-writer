# 创业计划书撰写 Skill

端到端引导用户完成一份专业、学术规范的创业计划书。
所有步骤必须严格执行，不可跳过。

---

## 执行总览

```
Step Pre-flight: 自动安装 → 检测软依赖技能，缺失则一次性全部安装
Step 0: 加载规范 → 读取要求文件或切换全球模板
Step 1: 项目定位 → 挖掘/引导用户明确创业方向（superpowers:brainstorming 增强）
Step 1.5: 设计风格 → StyleKit 推荐风格 + 用户选择 + 提取设计 Token（ui-ux-pro-max 协作）
Step 2: 产品打磨 + 数据调研 → 问题定义→用户洞察→分析即调研（AI维护调研记忆）→定价→获客→路线图→数据闸门→CEO审查
Step 3: 技术方案 → 三级分流（完整/轻量/跳过）+ 五阶段三引擎协作 + 安全门禁（基于已验证数据做架构决策）
Step 4: 撰写输出 → 对标学习标杆→从调研记忆取数→逐章撰写→HTML生成→去AI痕迹
Step 5: 质量检查 → 答题板核查 + superpowers:verification-before-completion 验证
Step 6: 路演答辩 → PPT 大纲（商业路演标准）+ 答辩陈词 + 常见问题预演
```

每完成一步，向用户汇报进展，确认后进入下一步。

**严禁跳过 Step 0-2 直接写计划书。** 没有充分铺垫的计划书是浪费时间。Step 2 完成时必须通过数据充分性闸门——所有分析有真实数据支撑，所有数据存入调研记忆。

### 0.2 加载规则

### 质量对标——两份标杆清单

创业计划书写到什么水平才算好？不能凭感觉。以下两套标杆全部公开可查、可检索、可学习——一套用来说服投资人（Pitch Deck），一套用来说服自己和团队（完整计划书）。Pitch Deck 教你怎么讲故事，完整计划书教你怎么把事情想清楚。

#### 标杆一：路演 PPT 对标（学怎么讲故事）

这些是拿到真金白银的 pitch deck，短小精悍，每页在卖一个观点。供 Step 6 路演 PPT 参考。

| 公司 | 阶段 | 结果 | 核心特征 |
|------|------|------|---------|
| Airbnb 2008 Seed Deck | 种子轮 $60万 | → $1000亿+ | 问题/方案一页说清，只讲最重要的三件事 |
| Uber 2008 Pitch Deck | 种子轮 $20万 | → $1500亿+ | TAM 计算和市场推演是教科书级 |
| Dropbox 2007 YC 申请 | YC 孵化 | → $100亿+ | 最朴素的语言说清产品，零 jargon |
| LinkedIn 2004 B 轮 | B轮 $1000万 | → 被微软 $262亿 收购 | 零收入靠愿景融资——"概念型路演"范本 |
| YouTube 2005 Pitch Deck | A轮 $350万 | → 被 Google $16.5亿 收购 | 在宽带都不普及的年代讲清视频平台未来 |

**检索方式：** WebSearch 搜 "{公司名} original pitch deck" 或 "{公司名} seed deck PDF" 即可找到原文件。

#### 标杆二：完整创业计划书对标（学怎么把事情想清楚）

Pitch deck 只有 10-14 页，是给投资人 10 分钟看的。你要写的是 15,000-25,000 字的完整计划书——结构、深度、数据密度完全不是一回事。以下是可以检索到、可以逐章学习的完整计划书标杆。

**一、SEC S-1 招股书（事实上的完整商业计划书）**

美国公司上市必须向 SEC 提交 S-1 文件。S-1 的"Business"章就是一份经过律师、投行、审计师反复拷问的完整商业计划书——不是学生作业，是融了数百亿美元的真东西。全部在 SEC EDGAR（sec.gov/edgar）免费公开，WebSearch 搜 "{公司名} S-1 SEC" 即达。

| 公司 | S-1 时间 | 融资金额 | 为什么值得学 | 重点章节 |
|------|---------|---------|-------------|---------|
| **Google 2004 S-1** | 2004.04.29 | $16.7亿 → $2万亿+ | Larry & Sergey 的"An Owner's Manual"创始人公开信，是所有商业计划书写作的最高标杆——坦诚、有远见、不废话。同时展示了双股权结构、荷兰式拍卖 IPO 等机制创新 | 创始人公开信、风险因素（坦诚度标杆）、商业模式 |
| **Facebook 2012 S-1** | 2012.02.01 | $160亿 → $1万亿+ | 移动转型叙事：一家靠网页起家的公司如何让投资人相信它能在移动端活下来。Zuckerberg 公开信定义了"Hacker Way"企业文化。风险因素列出了 50+ 条具体风险——不是"政策风险"这种泛词 | 创始人公开信、风险因素（长达 20 页，每条可操作）、业务描述 |
| **Alibaba Group 2014 S-1** | 2014.05.06 | $250亿（史上最大 IPO） → $3000亿+ | **中国学生最该读的标杆。** 300+ 页，中英对照语境。Jack Ma 的"致股东信"解释了中国互联网公司的独特生态。电商平台的双边市场逻辑、VIE 结构说明、支付宝的监管风险——所有中国创业公司面临的特殊问题都能在这里找到怎么写 | 创始人公开信、业务章节（平台生态描述）、风险因素、VIE 结构说明 |
| **Tesla 2010 S-1** | 2010.01.29 | $2.26亿 → $1万亿+ | 亏损公司怎么上市——零利润但有 1.09 亿美元收入，靠的是讲清楚"为什么现在亏钱是对的"。硬科技公司的商业计划书范本 | 业务描述、风险因素、财务数据（亏损但逻辑清晰） |
| **Uber 2019 S-1** | 2019.04.11 | $81亿 → $1500亿+ | 双边市场如何描述飞轮效应。平台型公司怎么写司机和乘客两边的关系。争议中上市的公司怎么写风险因素——Uber 的风险因素写到了"如果我们不能盈利怎么办" | 平台网络效应描述、风险因素（坦诚度标杆）、财务数据 |

**二、商业计划书竞赛获奖作品**

以下比赛每年公开获奖作品，是学生创业计划书的最直接对标——和你在做一样的事情，拿了奖，融到了钱。

| 比赛 | 主办方 | 为什么值得学 | 检索方式 |
|------|--------|-------------|---------|
| **Rice Business Plan Competition** | Rice University | 全球最大、奖金最高的研究生创业计划赛。获奖团队共融资超过 $50 亿。获奖 BP 结构完整、财务模型精细 | WebSearch: "Rice Business Plan Competition winners business plan" |
| **MIT $100K Entrepreneurship Competition** | MIT | 1990 年创办，参赛者中诞生了 250+ 家公司，融资超 $20 亿。技术型创业计划书的最佳参照 | WebSearch: "MIT 100K competition winning business plan" |
| **Harvard Business Plan Contest (HBS New Venture Competition)** | Harvard Business School | 上场就是 VC 在评。获奖 BP 的商业逻辑严密、市场分析有独特框架 | WebSearch: "Harvard Business School New Venture Competition winner business plan" |
| **互联网+ 大学生创新创业大赛 金奖** | 中国教育部 | 中国最大规模，金奖作品代表了国内比赛的最高标准。注意：比赛型 BP 的典型风格——框架完整、数据丰富但核心洞察不一定尖锐 | WebSearch: "互联网+ 大赛 金奖 创业计划书" 或 "互联网+ 创新创业大赛 获奖作品" |
| **挑战杯 大学生创业计划竞赛** | 共青团中央/中国科协/教育部 | 国内历史最悠久的大学生创业比赛。金奖作品可作格式和结构参照 | WebSearch: "挑战杯 创业计划竞赛 金奖作品" |

**使用方式：**
1. 搜到获奖 BP 原文或摘要后，重点看三样东西：**目录结构**（他们怎么组织章节的）、**执行摘要**（他们怎么在 2 页内把故事讲完整）、**财务模型**（他们用什么格式呈现数据）
2. 不要照抄结构——比赛型 BP 和真创业 BP 的区别见下方
3. 同一项目对标 2-3 份不同比赛的获奖 BP，交叉参照

**比赛型 BP vs 真创业 BP 的本质区别：**
- 比赛型 BP：框架完整、面面俱到、看起来专业——读完记不住核心论点
- 真创业 BP：**一个核心洞察贯穿全文**、对单位经济模型有偏执般的细节、诚实面对风险、有具体的而非泛泛的进入市场路径
- 比赛型 BP 追求"不扣分"，真创业 BP 追求"让读者记住一件事并举出证据"
- **本 skill 的目标：写出结构上满足比赛标准、内容上接近真创业 BP 水准的计划书**

**撰写时牢记三条原则：**
1. 每段话必须为你的核心论点添砖加瓦，否则删掉
2. 每个数据必须解释它意味着什么机会或风险，不堆数字
3. 诚实 > 完美。投资人（和老师）一眼能看出你在回避什么

> **注意：** 美团和字节跳动的早期 BP 未经公开（字节 2012 年天使轮甚至没有正式 BP，是纸巾上的草图）。本 Skill 不引用未经证实的历史案例。

然后按照 13 章模板撰写：
封面 → 目录 → 执行摘要 → 企业介绍 → 产品与服务 → 行业与市场分析 → 营销计划 → 管理团队 → 运营计划 → 财务计划 → 风险分析 → 退出策略 → 附录

### 0.3 文档预处理

- **`.docx` 文件**：`npx mammoth "文件路径"` 提取文本
- **`.doc` 文件（OLE2 格式）**：mammoth 无法处理。使用 Node.js 读取原始字节，扫描 UTF-16LE 编码提取中文文本（Unicode 范围：0x4E00-0x9FFF 汉字，0x3000-0x303F 标点，0xFF00-0xFFEF 全角符号）。核心代码模式：
  ```javascript
  const buf = fs.readFileSync(path);
  let text = '';
  for (let i = 0; i < buf.length - 1; i += 2) {
    const code = buf.readUInt16LE(i);
    if ((code >= 0x4E00 && code <= 0x9FFF) || (code >= 0x3000 && code <= 0x303F) ||
        (code >= 0xFF00 && code <= 0xFFEF) || (code >= 0x0020 && code <= 0x007E) ||
        code === 0x000A || code === 0x000D) text += String.fromCharCode(code);
  }
  ```
- **`.pdf` 文件**：使用 Read 工具直接读取，或 `npx pdf-parse "文件路径"`
- 如果 mammoth 不可用，提示运行：`npm install -g mammoth`

## Step 1: 项目定位

### 1.0 头脑风暴（先发散，后收敛）

在进入结构化问答之前，先用开放式对话帮用户理清思路。大多数人说不清自己的创业想法是因为还没被问对问题。

**Step 1.0.1: 调用 /office-hours**

用 Skill 工具调用 `office-hours`。该 skill 会根据用户状态自动选择 Startup Mode（YC 产品诊断六问）或 Builder Mode（设计合伙人）。

将 office-hours 的输出（问题诊断、用户原话、设计文档路径）存入工作记忆——这些是第一手素材，Step 1.2 的结构化问答和后续所有章节都从这里取材。

**Step 1.0.2: 脑暴结果沉淀**

office-hours 结束后，用一段话总结：
- 用户真正想解决的问题是什么（不是他们说的第一句话，而是被追问后浮现的那个）
- 目前的解决方案是什么（哪怕是用 Excel + 微信群拼凑的）
- 用户对这个项目的野心有多大（课余作业 / 真想创业 / 还没想清楚）

向用户确认这段总结，然后进入 1.1。

**降级策略：** 如果 office-hours 不可用，执行内置脑暴三问：
1. "你上一次亲身遇到这个问题是什么时候？当时你是怎么解决的？花了多少时间/钱？"
2. "如果有人已经在做这件事了，你觉得他们做得最差的地方是什么？"
3. "如果这个项目失败了，最可能的原因是什么？"

三问结束后，总结沉淀，进入 1.1。

### 1.2 用户有明确方向

**问题 1/4（D1）：产品/服务类型**
> 你的产品/服务最接近以下哪种形态？

选项：

- B) 硬件/实体产品 — 需要生产制造和供应链管理
- C) 线下服务 — 餐饮、教育、咨询、养老等

- A) 大学生/年轻人 — 消费降级、求职焦虑、学习效率、社交孤独

选项：
- A) 有相关行业实习或工作经验
- B) 有技术能力（编程、设计、工程等）
- C) 有市场渠道或人脉资源
- D) 有深度研究积累（论文、课题、专利等）

**问题 4/4（D4）：当前阶段**

- A) 只有想法，还没动手
- B) 正在做用户调研或需求验证
- C) 已有 MVP/原型，开始获取早期用户

   - 适合大学生的切入点

## Step 1.5: 设计风格选择（StyleKit + ui-ux-pro-max 协作）

在产品打磨之前，先确定设计方向。这样后续所有产出（产品原型、HTML 计划书）都有一致的视觉语言。

### 1.5.1 StyleKit 访问

1. `WebFetch` 读 `https://stylekit.top/llms.txt` 获取最新风格目录
2. 不可用时用 `references/stylekit-styles.md` 内置速查表
3. 选定后 `WebFetch` 读 `https://stylekit.top/api/styles/{slug}` 获取 Token 包

### 1.5.2 风格推荐（动态发现 + 五轮推荐制）

**不使用固定的项目类型→风格映射表。** 固定映射会让同类项目生成相似的视觉产出，千人一面。
改为每次根据实际项目特点和当前创业趋势动态推荐，确保每个方案都有独特的视觉个性。

**Step A: 感知项目气质**

根据 Step 1 收集的信息，提炼 3 个关键词描述项目的品牌气质。
例如：教育类→"温暖、清朗、信任" vs AI工具类→"锋利、极简、未来感"。
这些关键词用于匹配 StyleKit 风格。

**Step B: 五轮推荐**

每轮用 AskUserQuestion 展示 3 个风格方案（最多 5 轮，共 15 个方案），每个方案包含：
- 风格名称（StyleKit slug）
- 2-3 个词的视觉关键词
- 一句解释为什么该风格匹配用户的项目气质
- 一个类似感觉的知名品牌参考

选项格式：
- A) `[slug]` — [视觉关键词] — [为什么匹配]
- B) `[slug]` — [视觉关键词] — [为什么匹配]
- C) `[slug]` — [视觉关键词] — [为什么匹配]
- D) 都不满意，看下一轮（或让 AI 帮你选最合适的）

**每轮 3 个方案必须来自不同的风格类别**（如现代科技/自然舒适/粗野主义/品牌灵感等），确保多样化。

从内置参考文件 `references/stylekit-styles.md` 获取可用的 120+ 风格清单和分类。

### 1.5.3 用户确认与引导

- 用户选 A/B/C → 确定风格，进入 1.5.4 提取设计资源
- 用户选 D 且未满 5 轮 → 询问是看下一轮还是让 AI 直接推荐；如看下一轮则展示 3 个全新方案（不与之前重复）
- 用户选 D 且已满 5 轮 → 进入引导模式：
  > "看来你对审美有自己清晰的想法。能描述一下你希望读者看到这份计划书时，第一眼感受到的是什么？比如专业严谨、温暖亲切、大胆前卫、还是简洁清晰？"
  
  根据用户描述的关键词，从 StyleKit 120+ 风格中手动匹配 2-3 个最接近的方案做最终确认。
- 用户选 D 且要求 AI 推荐 → 根据 Step A 提炼的品牌气质关键词自动选择最佳匹配

### 1.5.4 提取设计资源

用户选定风格后：

1. 用 `WebFetch` 读取 `https://stylekit.top/api/styles/{slug}` 获取：
   - 颜色调色板（Primary/Secondary/Accent/Neutral/Semantic）
   - CSS 变量或 Tailwind Token
   - 字体方案（标题字体 + 正文字体）
   - 阴影层级
   - 圆角规格
   - 组件示例代码
2. 将 Token 存入本次工作上下文，供 Step 3 原型和 Step 4 HTML 使用

如果 API 不可达，使用内置的 StyleKit 风格参考卡（`references/stylekit-styles.md`）。

### 1.5.5 StyleKit 与 ui-ux-pro-max 分工

StyleKit 提供设计 Token（颜色/字体/阴影/圆角），ui-ux-pro-max 做品牌深化和视觉品质提升。两者不重叠——先取 Token，再精加工。

## Step 2: 产品打磨 + 数据调研

Step 2 的核心原则：**分析即调研。** 不做没有数据支撑的分析，不写没有来源出处的结论。每个分析模块产出结论的同时，产出支撑结论的真实数据——数据不是单独"搜"出来的，是在分析过程中自然沉淀下来的。

### 调研记忆系统

在整个 Step 2 过程中，AI 在上下文内维护一份结构化的**调研记忆**。这不是写到磁盘的文件——是 Step 4 撰写时的唯一数据源。写完 Step 2，调研记忆就是一本完整的数据手册。

**记忆结构（对话上下文中维护）：**

```
📊 调研记忆
═══════════════════════════════════════

[数据点]
- {具体数字/事实}
  来源：{机构/报告名称}
  URL：{完整链接}
  检索：{YYYY-MM-DD}
  用于：{支撑什么分析/哪个章节}

[学术文献]
- {作者}. {标题}[{文献类型}]. {期刊/出版社}, {年份}, {卷期}: {页码}.
  来源：{知网/万方/Google Scholar}
  用于：{引用位置}

[产品决策]
- {决策内容}
  理由：{为什么选这个}
  数据支撑：{哪个数据点支持了这个决策}
  替代方案：{考虑过但否决的方案及否决原因}
```

**每次分析结束后更新。** 完成一个分析模块（如 TAM/SOM/SOM、竞品分析、定价策略），立即将本模块产出的数据、文献、决策追加到调研记忆中。不等到 Step 2 结束才统一整理——分批写入，避免遗忘。

**禁止编造。** 搜不到精确数据时用区间估算，标注"[估算]"和估算依据。找不到的数据说"未找到公开数据"——不编。

### 2.0 品牌体系设计

在进入市场分析之前，先帮用户完成品牌基础建设。品牌体系是后续产品打磨的基石，也直接影响 Step 1.5 的设计风格选择。

#### 2.0.1 产品命名与商标

用 AskUserQuestion 引导用户明确：
- 产品/品牌名称（如已确定，确认其含义和故事）
- 商标注册情况（已注册/申请中/未申请/尚未考虑）
- 是否需要设计 Logo 概念

#### 2.0.2 品牌故事与 Slogan

引导用户提炼：
- **品牌故事**：为什么做这件事？品牌的起源和初心是什么？
- **品牌 Slogan**：一句话传达品牌核心价值
- **品牌调性关键词**：3-5 个描述品牌气质的词（如"温暖、可信、智慧"），这些词将影响 Step 1.5 的风格选择

#### 2.0.3 品牌视觉基础（VI 概要）

在 Step 1.5 StyleKit 风格选定后，基于选定的设计 Token，帮用户定义：
- 主色调与辅助色（与 StyleKit Token 保持一致）
- Logo 概念描述（形状、意象、风格）
- 品牌应用场景清单（App 图标、Web 页面、宣传物料等）

#### 2.0.4 问题定义（先定义问题，再谈方案）

在进入市场分析和功能挖掘之前，先用结构化 PM 工具把问题定义清楚。问题定义是所有后续分析的根基——问题定义错了，后面全偏。

**调用 `problem-statement` 或 `problem-framing-canvas`：**

用 Skill 工具尝试调用。两者至少调用一个：

| 优先级 | Skill 名称 | 强制回答的问题 | 不可用时的替代方案 |
|--------|-----------|--------------|------------------|
| 1 | `problem-statement` | 为谁解决什么问题？现有方案为什么不够好？成功标准是什么？问题发生的频率和代价？ | 内置问题框架问答（5 问） |
| 2 | `problem-framing-canvas` | 问题发生的场景和触发条件？谁受这个问题影响最大？他们目前在怎么应付？ | 自建问题画布（六格模板） |

**内置问题框架（Skill 不可用时）：**

1. **问题是什么？** 一句话描述。如果是"XX不方便"这种模糊表述，追问到具体场景——"上次这个不方便让你损失了什么？"
2. **谁的问题？** 受影响的群体有多大？其中谁最痛（early adopter）？
3. **他们现在怎么解决的？** 竞品、替代方案、或者就忍着？如果用户说"没有替代方案"，追问："他们现在是不是用 Excel + 微信群拼凑着？"
4. **为什么现有方案不够好？** 竞品/替代方案最让用户抓狂的三个点是什么？
5. **你怎么知道问题解决了？** 成功的衡量标准是什么？（不是"用户满意度提升"，是"用户在 3 分钟内完成订单"这种可观测的指标）

产出：**问题定义陈述**（≤200 字），作为后续所有分析的北极星。向用户确认后进入 2.1。

### 2.1 用户洞察与功能挖掘

#### 2.1.0 用户发现（先听后想）

大多数学生没做过真正的用户调研。直接跳到功能挖掘等于在沙滩上盖楼。本步骤帮用户建立最基础的用户发现能力。

**调用 `discovery-process` 或 `discovery-interview-prep`：**

| 优先级 | Skill 名称 | 用途 | 不可用时的替代方案 |
|--------|-----------|------|------------------|
| 1 | `discovery-process` | 完整的用户发现流程指导 | 内置发现三问 |
| 2 | `discovery-interview-prep` | 访谈提纲设计、避免引导性问题、洞察提取 | 内置访谈指南 |

**内置发现三问（Skill 不可用时）：**

1. "你上次亲身遇到这个问题是什么时候？当时你怎么解决的？花了多少时间/钱？"
2. "你能不能这周找 3 个目标用户聊一聊？问他们：'你现在怎么处理这件事的？最让你烦的是什么？' 不要问他们要不要你的方案，只问他们现在有多痛。"
3. "有没有什么人已经在用自己的土办法解决这个问题了？（这些人是你最应该关注的 early adopter）"

产出向用户确认后，带着真实的用户痛感进入功能挖掘。

#### 2.1.1 功能挖掘与优先级

在用户描述了初始产品想法后，用 AskUserQuestion 逐轮深挖。目标是事无巨细——用户心里模糊的想法，通过提问逼出清晰的答案。每轮一个主题，不跳不混。

**第一轮：功能补充**
> 除了你刚才描述的核心功能，你还希望产品具备哪些差异化功能？有没有用户在使用过程中可能需要的辅助功能——比如通知提醒、数据统计、社交分享、收藏夹这类？先不设限，想到什么说什么。

**第二轮：优先级排序**
> 如果按 MVP（最小可行产品）、V1.0（首版正式发布）、V2.0（后续迭代）三级来分，你会把刚才提到的功能分别归入哪一级？P0 必须有什么？P1 规划什么？P2 远期什么？

**第三轮：盈利模式细节**
> 你设想的主要收入来源是什么？具体怎么收——按比例抽佣还是固定服务费？抽多少？定价依据是什么？有没有补贴或免费策略来拉新？除了主营收入，是否考虑过广告、增值服务、数据服务、API 开放等其他变现方式？

**第四轮：运营与服务**
> 产品上线后的日常运营怎么做？用户下单后谁来处理物流/配送（自建还是合作）？客服由谁负责、响应时效什么样？遇到交易纠纷（退货、投诉、差评）怎么处理？有没有争议仲裁机制？

**第五轮：增长策略**
> 第一批用户从哪里来？你打算通过什么渠道拉新（校园地推、社团合作、社交媒体、SEO、口碑推荐）？怎么让用户留下来而不是用一次就走？有没有病毒传播机制——比如邀请奖励、拼团、分享得优惠？

**第六轮：国际化与扩展**
> 产品未来是否需要考虑国际化？如果要出海，先在哪些市场推出？产品名称、品牌、功能架构是否方便多语言和多币种扩展？如果暂不国际化，国内是否有多城市扩展计划？

#### 2.1.2 技能检测与调用

检测 pm-skills 是否可用。尝试用 Skill 工具调用以下技能（按顺序）：

**基础分析（必须调用）：**

| 2 | `tam-sam-som-calculator` | 计算市场规模 TAM/SAM/SOM | WebSearch + 手动构建模型 |

| 5 | `company-research` | 竞品公司深度研究（融资/团队/产品演进/战略动向） | WebSearch + 手动收集 |

| 7 | `business-health-diagnostic` | 商业模式健康检查 | 自建商业模式画布 |

| 13 | `prd-development` 或 `feature-investment-advisor` | PRD 文档与功能投资优先级 | 问答引导 + 自建 PRD 模板 |
| 14 | `prioritization-advisor` | 功能优先级排序（RICE/ICE/Kano 等框架） | 手动 MVP/V1/V2 分级 |
| 15 | `brand-strategy` 或 `naming-brainstormer` | 品牌策略与产品命名 | 品牌体系设计问答 |

**场景化技能（按项目类型选择性调用）：**

| 项目类型 | 推荐技能 | 用途 |
|---------|---------|------|
| SaaS/软件/平台 | `saas-economics-efficiency-metrics` | 效率指标（burn multiple、Rule of 40、毛利率基准） |
| SaaS/软件/平台 | `saas-revenue-growth-metrics` | 收入增长指标（NDR、ARPU、expansion revenue） |
| AI 驱动型 | `ai-shaped-readiness-advisor` | AI 就绪度评估（数据护城河、模型壁垒、AI 定价模式） |
| 消费品牌/内容 | `storyboard` | 用户场景故事化——把关键体验画成故事板，找到情感触点 |
| 双边平台 | `lean-ux-canvas` | 补充 BMC 对用户体验和双边网络效应的盲区 |

**内置 JTBD 四问（`jobs-to-be-done` 不可用时）：**

1. **功能任务：** 用户需要完成什么具体任务？（例如："我需要快速知道这门课值不值得选"——不是"课程评价系统"）
2. **情感任务：** 用户在使用过程中想感受到什么？想避免什么感受？（例如："不想选完课才发现老师给分很低，那种后悔感"）
3. **社会任务：** 用户使用这个产品时，别人怎么看他们？（例如："室友都觉得我很会选课"）
4. **替代方案：** 如果没有你的产品，用户目前是怎么完成这个任务的？用了多少时间/钱/人情？

**内置发现三问（`discovery-process` 不可用时，在 2.1.0 已执行）：**

**调用规则：**
- 每个 Skill 调用后，将其输出整理为结构化内容，存入工作记忆
- 如果 Skill 不可用（报错/未找到），立即切换到替代方案，不要反复尝试
- 基础分析（1-7）尽量全部覆盖；进阶分析（8-15）根据项目复杂度选择性调用；场景化技能仅在项目类型匹配时调用
- 每个分析环节完成后，向用户展示摘要

### 2.2 核心分析产出（不可跳过）

无论使用 Skill 还是替代方案，以下内容**必须**产出。每个分析模块同时产出结论和数据——数据存入调研记忆。

1. **问题定义陈述**（≤200 字，来自 2.0.4）——为谁解决什么问题，现有方案为什么不够好，成功标准是什么
2. **JTBD 用户任务分析**（来自 2.1.2）——功能任务、情感任务、社会任务、当前替代方案
3. **TAM/SAM/SOM 数据**（三条线，有具体数字，数据来源标记 URL）
   → 🔍 搜："{行业} 市场规模 2025 2026" / "{行业} 市场规模 CAGR 预测"
4. **目标用户画像**（2-3 个 Persona，包含人口统计、行为特征、痛点、JTBD）
   → 🔍 搜："{目标人群} 消费行为 调查报告 2025" / "{目标人群} 用户画像 2025"
5. **竞争格局图**（至少 3 个竞品深度对比——不只是名字，含融资/产品/商业模式/用户评价细节）
   → 🔍 搜：每个竞品的融资轮次/金额/估值、App Store 评分、用户投诉关键词
6. **商业模式画布**（9 要素完整版）
7. **SWOT 分析**（至少每个维度 3 个要点）
   → 🔍 搜："{行业} 政策 2025 2026 十四五" / "{行业} 监管政策 最新"
8. **定价策略文档**（定价模型、竞品价格对比、用户支付意愿分析、盈亏平衡点——见 2.6）
   → 🔍 搜：竞品定价（官网/App Store）、"{目标人群} 付费意愿 调研"
9. **获客渠道策略**（首选渠道、CAC 估算、可扩展性评估、各阶段渠道组合——见 2.7）
   → 🔍 搜："{渠道} 获客成本 行业基准 2025" / "{行业} CAC LTV 基准"
10. **财务对标数据**（2-3 家可比公司的实际经营数据：收入规模、成本结构、毛利率、研发费用率）
    → 🔍 搜：可比公司财报/招股书/行业研报；找不到竞品则用同行业上市公司年报

**每个 🔍 标记的搜索动作不可跳过。** 搜索结果同步写入调研记忆的 [数据点] 区。

### 2.3 数据质量要求

- 所有市场数据必须来自最近 3 年
- 数据来源优先级：政府统计 > 行业报告 > 咨询公司 > 媒体报道
- 禁止编造数据。如果搜不到精确数据，使用区间估算并标注"估算依据"

### 2.4 PRD 文档生成（技术型项目必须）

对于涉及软件/App/平台开发的项目，在进入技术方案（Step 3）之前，必须先产出产品需求文档（PRD）。PRD 是技术方案的输入，避免开发混乱。

**PRD 必须包含：**

1. **产品概述**：一句话描述产品是什么，为谁解决什么问题
2. **用户故事**（User Stories）：至少 5 个核心场景，按"作为[角色]，我希望[功能]，以便[价值]"格式
3. **功能需求**（Functional Requirements）：按优先级排列
   - **P0（MVP 必须）**：最小可行产品必须包含的功能，没有它产品无法运转
   - **P1（V1.0 规划）**：首版正式发布应包含的功能，构成完整用户体验
   - **P2（后续迭代）**：远期规划功能，可根据市场反馈调整
4. **非功能需求**：性能指标、安全要求、可扩展性、兼容性等
5. **产品路线图**：MVP → V1.0 → V2.0 的时间节点和里程碑
6. **国际化策略**：是否支持多语言/多币种/跨境？如是，优先级和阶段

**PRD 产出后，向用户展示摘要，确认功能优先级和路线图，再进入 Step 3。**

如果 `prd-development` 或 `feature-investment-advisor` Skill 可用，调用它们辅助生成。否则用问答引导 + 自建模板完成。

### 2.5 产品路线图（技术型/产品型项目推荐）

PRD 定义了"做什么"，路线图定义"什么时候做、依赖什么、需要什么资源"。仅功能清单的三级分类不够——投资人想看的是你对执行节奏的判断力。

**调用 `roadmap-planning`：**

| 优先级 | Skill 名称 | 用途 | 不可用时的替代方案 |
|--------|-----------|------|------------------|
| 1 | `roadmap-planning` | 结构化路线图（里程碑/依赖/时间线/资源配置） | 自建路线图模板 |

**路线图必须包含：**

1. **时间线**：MVP → V1.0 → V2.0 的时间节点（具体到季度）
2. **里程碑**：每个阶段的关键交付物和成功标准
3. **依赖关系**：哪些功能必须在哪些功能之前完成？为什么？
4. **资源配置**：每个阶段需要什么角色、多少人、什么技能
5. **风险缓冲**：最可能延期的环节，以及怎么应对

**产出格式：** 一张带时间轴的路线图（Mermaid gantt 或表格均可）+ 每个阶段的 3-5 行说明（目标/产出/风险）。

如果 `roadmap-planning` 不可用，用内置模板完成。纯服务型/内容型项目可精简为 2-3 个里程碑。

### 2.6 定价策略深潜

定价是投资人和老师最盯着的三个问题之一。不是"我们打算收多少钱"一句话能回答的——要有逻辑、有对标、有验证。

**调用 `finance-based-pricing-advisor`：**

| 优先级 | Skill 名称 | 用途 | 不可用时的替代方案 |
|--------|-----------|------|------------------|
| 1 | `finance-based-pricing-advisor` | 价值定价/成本加成/竞争定价/freemium 分析/支付意愿 | 内置定价框架 |

**内置定价框架（Skill 不可用时，逐项完成）：**

1. **定价模型选择**：
   - 订阅制（月/年）— 适合 SaaS、内容平台
   - 交易抽佣 — 适合双边市场、电商
   - 一次性付费 — 适合工具、硬件
   - Freemium（免费基础 + 付费高级）— 适合需要网络效应的产品
   - 分层定价（Basic/Pro/Enterprise）— 适合 B2B
   - 混合模型 — 实际业务中最常见

2. **竞品价格对标**：
   - 3 个最接近竞品的定价（具体数字，来源 URL）
   - 你的定价比竞品高还是低？为什么？（高 → 价值差异在哪；低 → 能否持续）
   - 竞品有没有隐藏收费？（例如：基础版便宜但高级功能很贵）

3. **用户支付意愿验证**：
   - 目标用户现在为替代方案花多少钱？（包括时间成本折算）
   - 有没有人已经愿意为类似产品付费了？（找证据，不凭空假设）
   - 如果免费，谁来补贴？（广告主？付费用户？投资人的钱？）

4. **盈亏平衡点**：
   - 在预期定价下，需要多少付费用户/交易量才能打平成本？
   - 这个数字在目标市场的渗透率是多少？（<1% 且可达到 = 健康；>10% = 定价太低或市场太小）

5. **定价演进路径**：
   - 启动阶段（拉新优先）→ 成长期（收入优先）→ 成熟期（利润优先）的定价变化
   - 有没有涨价空间？（用户越离不开你，价格弹性越低——但这需要时间）

产出：**定价策略文档**（≥500 字），作为 2.2 核心产出的第 8 项。

### 2.7 获客渠道策略

"用户从哪里来"是创业计划书里最容易写得很乐观的部分。"在社交媒体上宣传"不是获客策略——渠道选择、CAC、可扩展性、各阶段组合，才是。

**调用 `acquisition-channel-advisor`：**

| 优先级 | Skill 名称 | 用途 | 不可用时的替代方案 |
|--------|-----------|------|------------------|
| 1 | `acquisition-channel-advisor` | 获客渠道结构化分析（付费/自然/病毒/销售驱动） | 内置获客框架 |

**内置获客框架（Skill 不可用时，逐项完成）：**

1. **渠道矩阵**：列出所有可能渠道，评估每个渠道的：
   | 渠道 | 目标用户浓度 | CAC 预估 | 可扩展性 | 适合阶段 | 优先级 |
   |------|------------|---------|---------|---------|--------|
   | （校园地推/社团合作/SEO/社交媒体/付费广告/口碑推荐/内容营销/合作伙伴/…） | 高/中/低 | ¥X | 有限/可扩展/高度可扩展 | 冷启动/增长/成熟 | P0/P1/P2 |

2. **首选渠道深挖**：选优先级最高的 1-2 个渠道，详细说明：
   - 为什么这个渠道最适合你的阶段和用户？
   - 具体怎么执行？（不是"做内容营销"，是"每周在知乎回答 3 个 XX 相关的问题，每篇带产品链接"）
   - 第一波 100 个用户怎么来？第二波 1000 个怎么来？

3. **CAC 与 LTV 匹配**：结合 2.6 的定价，
   - 预估 CAC（每个渠道分开估算）
   - 预估 LTV（基于定价 × 预期留存时间）
   - **LTV/CAC ≥ 3：1 是及格线**。如果达不到，要么改渠道，要么改定价。

4. **病毒/推荐机制**（如适用）：
   - 有没有内置的传播机制？（邀请奖励、拼团、分享得优惠）
   - 病毒系数预估：一个用户平均能带来几个新用户？
   - 如果病毒系数 < 0.5，不要指望靠病毒传播——老老实实做付费获客。

5. **获客时间线**：
   - 启动期（0-3 个月）：主要靠什么？目标获客数？
   - 增长期（3-12 个月）：渠道怎么扩展？CAC 怎么优化？
   - 规模期（12 个月+）：什么时候渠道饱和了？下一个增长引擎是什么？

产出：**获客渠道策略文档**（≥500 字），作为 2.2 核心产出的第 9 项。

### 2.8 CEO 视角审查（gstack /plan-ceo-review）

在进入技术方案（Step 3）之前，用 CEO 视角对整个商业逻辑做一次审视。问题不在技术——在商业逻辑有没有漏洞。

**用 Skill 工具调用 `plan-ceo-review`**（gstack 技能）。

如果可用：将 2.2 的全部核心产出提交给 `/plan-ceo-review`，它从 CEO 视角审视：
- 这个生意的单位经济模型能不能成立？
- 收入和成本的增长曲线能不能交叉？
- 最脆弱的一环在哪？
- 有没有"看起来很赚钱但实际不能规模化"的陷阱？

如果 `/plan-ceo-review` 不可用，执行内置 CEO 审查五问：

1. **"这个生意的单位经济模型，三年内能不能打平？"** 用 2.6 的定价和 2.7 的 CAC 算清楚——每获得一个用户，赚回来的钱（LTV）是不是大于获得这个用户花的钱（CAC）× 3？
2. **"如果今天就要倒闭了，最可能的原因是什么？"** 这是投资人的必问题。答案应该在 2.2 的 SWOT 和风险分析里能找到对应——没找到说明 SWOT 写得太客气了。
3. **"竞品今天决定免费做你做的事，你怎么办？"** 你的护城河到底是什么——不是"更好的用户体验"这种万能答案，是网络效应/数据飞轮/独家供应链/监管牌照/专利。
4. **"你的团队为什么能做成而别人不行？"** 因为技术能力？行业经验？渠道资源？还是只是"我们先想到了"——"先想到"不是壁垒。
5. **"砍掉 50% 的功能，这个生意还能成立吗？"** 如果能——那些被砍的功能就是噪音，在 MVP 里删掉。如果不能——说明核心功能就是那 50%，剩下的砍掉。

**不通过不进入 Step 3。** 发现逻辑漏洞回到对应步骤修正——不是推倒重来，是定位到具体哪条假设不成立，修正它。

**gstack 竞品实测（可选但推荐）：**

如果竞品有可公开访问的产品（App/网站），用 `browse` 技能实际操作竞品产品 10 分钟，记录：
- 注册/激活流程有几个步骤？有没有摩擦点可以利用？
- 核心功能的实际体验怎么样？（不是 PR 稿里写的，是你亲手点的）
- 定价页有没有隐藏条款？（免费试用后自动扣费？基础版功能阉割到不可用？）
- 竞品产品的"Aha moment"在第几步？（你用了多久才觉得"这东西有用"？）

这些实测笔记直接写入 2.2 竞争格局图——比任何研报都真实。

## Step 3: 技术方案

### 3.1 适用性判断（三级分流）

根据项目技术含量，走不同深度的技术方案路径：

**路径 A — 完整技术方案（五阶段全流程）：**
适用于：软件/SaaS/App/平台类、硬件/智能设备类、需要研发的技术驱动型项目。
→ 执行 3.2-3.7 全部。

**路径 B — 轻量级技术方案（仅架构简述）：**
适用于：混合型项目——主营线下服务但有线上组件（如餐饮+小程序、零售+App、咨询+预约系统）。
→ 跳过四阶段编码，只做 3.2-B 轻量拆解 + 3.7 内置流程，控制在 1 轮对话内完成。

**路径 C — 跳过技术方案：**
适用于：纯服务型（线下餐饮、咨询、零售等）、纯内容/IP 型项目。
→ 在计划书中用 1-2 段简述"技术实现"即可，无需深入架构和编码。

用 AskUserQuestion 确认用户项目属于哪条路径（推荐判断 + 让用户确认）。

### 3.2 PRD 需求拆解与实现方案选择

Step 2.4 产出的 PRD 文档是技术方案的唯一输入。

**Step 3.2.1: 需求模块拆解**

将 PRD 中的 P0/P1 功能需求按技术模块拆解：

| 模块 | 范围 | 示例（以校园二手书平台为例） |
|------|------|---------------------------|
| 前端/客户端 | 用户直接交互的部分 | 小程序 UI、Web 管理后台、搜索/浏览/下单流程 |
| 后端/API | 业务逻辑与数据 | 用户系统、交易引擎、库存管理、支付集成 |
| AI/算法 | 智能功能 | 定价引擎、推荐系统、图像识别（ISBN扫描） |
| 基础设施 | 部署与运维 | 云服务器、数据库、CDN、监控告警 |

每个模块列出其依赖关系和优先级。

**路径 B 在此处停止**——拆解完成后直接产出简化版：技术栈选型（一句话）+ 架构草图（一张 Mermaid）+ 开发路线图（3 个里程碑），然后进入 Step 4。

**Step 3.2.2: 实现方案生成**（仅路径 A）

针对拆解后的需求，产出 2-3 套实现方案供用户选择。每套方案包含：

| 维度 | 说明 |
|------|------|
| 技术栈 | 前端/后端/数据库/AI 的具体选型 |
| 开发周期 | MVP 多久，V1.0 多久 |
| 团队需求 | 需要什么角色、各多少人 |
| 成本估算 | 云服务 + 第三方 API + 人力成本（月/年） |
| 主要风险 | 技术风险 + 应对措施 |
| 适用场景 | 什么情况下选这套 |

**方案模板：**
- **方案 A（极简验证）**：最少资源最快上线。低代码/无代码优先，第三方服务最大化复用。
- **方案 B（标准工程）**：平衡速度与质量。主流框架 + 云服务，为后续迭代留架构空间。
- **方案 C（极致架构）**：最佳长期可维护性。微服务/领域驱动/自研关键模块。（技术壁垒型项目适用，非必出）

用 AskUserQuestion 让用户选择方案，确认后进入 3.3。

### 3.3 三引擎协作模型

技术方案阶段采用 **superpowers（流程）+ gstack（工程）+ mattpocock（代码品质）** 三引擎互补：superpowers 管怎么做对，gstack 管具体怎么做，mattpocock 管代码怎么写好。详细映射见 `references/three-engine-workflow.md`。

### 3.4 分阶段执行（五阶段，含安全门禁）

详细流程见 `references/three-engine-workflow.md`。概要：

| 阶段 | 主导引擎 | 关键产出 | 移交物 |
|------|---------|---------|--------|
| 一：技术规划 | superpowers + mattpocock | 技术选型矩阵、系统边界、ADR（≥3条）、风险点 | 规划移交摘要 |
| 二：架构设计 | gstack + mattpocock | 架构图（Mermaid）、API 设计、数据模型、安全自查清单 | 架构移交摘要 |
| 三a：原型验证 | mattpocock | 可运行原型（可丢弃），仅验证可行性，不含测试 | 原型验证记录 |
| 🔒 安全门禁 | — | 安全清单全通过方可进入三b | 安全门禁结果 |
| 三b：正式实现 | mattpocock + superpowers | TDD 开发、测试套件、生产级代码 | 实现移交摘要 |
| 四：评审验证 | 三引擎合流 | 评审报告、问题清单、修复记录、成本回溯验证 | 最终评审报告 |

**阶段移交摘要格式：** 每个阶段结束时写 3-5 行移交摘要，包含：
```
[阶段名称] 完成
决策：本阶段做了哪些关键选择（1-2 句）
产出：具体文件/图表/代码位置
风险：本阶段发现但留给下阶段解决的风险
```

**引擎不可用时：** 按 3.5 降级策略跳过该引擎步骤。全部不可用时用 3.7 全内置后备流程。

### 3.6 安全门禁检查

**触发条件：** 项目涉及以下任一场景时，在阶段三a 完成后、三b 开始前强制通过安全门禁：
- 用户注册/登录/身份认证
- 支付/交易/资金处理
- 用户个人数据收集与存储
- 第三方 API 集成（尤其是支付、地图、社交登录）
- 用户生成内容（UGC）

**安全清单：**

```
□ 认证安全：密码是否 bcrypt/argon2 哈希？是否支持 MFA？会话 Token 是否 httpOnly + Secure？
□ 数据安全：敏感数据是否加密存储？传输是否全量 HTTPS？是否有数据备份策略？
□ 隐私合规：是否收集了不必要的用户数据？是否有隐私政策？是否符合《个人信息保护法》？
□ API 安全：是否有 rate limiting？是否有 CSRF 保护？输入校验是否在服务端执行（不只是前端）？
□ 依赖安全：第三方 SDK/API 是否有已知漏洞？关键依赖是否固定版本？
□ 支付安全：是否使用成熟的支付网关（微信/支付宝/Stripe）而非自建？PCI-DSS 合规？
```

**不通过不进入三b。** 发现高危项（如明文存储密码、自建支付）必须修复。中低危项记录为已知风险，在三b 中处理。

**无安全场景的项目（纯静态展示、本地工具等）跳过此门禁。**

### 3.7 全内置后备流程

当三个引擎全部不可用时（路径 A）：

1. 绘制系统架构图（Mermaid）
2. 确定技术栈选型（前端/后端/数据库/部署/监控）
3. 撰写架构决策记录（ADR，≥3 条），格式：`标题、背景、决策、后果、替代方案`
4. 定义核心 API 和数据模型
5. 制定开发路线图（3-4 阶段：MVP → Alpha → Beta → GA）
6. 识别技术风险与缓解措施（至少 5 项）
7. 执行安全清单自查（如适用）
8. 给出开发团队与资源需求估算

## Step 4: 撰写与输出

### 4.0 撰写前对标学习（必须执行）

在写任何内容之前，先学习标杆。不清不楚地写，写出来的一定是平庸的。

**Step 4.0.1: 搜索并精读标杆计划书**

用 WebSearch 搜索以下标杆（至少选 3 个不同类型的，覆盖 S-1 招股书和竞赛获奖作品）：

```
搜索 SEC S-1（至少选 2 个）:
  WebSearch: "Alibaba 2014 S-1 SEC Edgar Business section"
  WebSearch: "Google 2004 S-1 SEC founder letter"
  WebSearch: "Facebook 2012 S-1 risk factors mobile"
  WebSearch: "Uber 2019 S-1 business description"

搜索竞赛获奖 BP（至少选 1 个）:
  WebSearch: "Rice Business Plan Competition winning plan executive summary"
  WebSearch: "MIT 100K business plan competition winner"
  WebSearch: "互联网+ 大学生创新创业大赛 金奖 创业计划书"
```

对每个搜索到的标杆，用 WebFetch 打开原文，精读：

| 精读重点 | 看什么 |
|---------|--------|
| **结构** | 章节怎么组织的？顺序怎么排的？执行摘要占了全书多大比例？附录里放了什么？ |
| **深度** | 数据引用密度多高？（每页/每千字几个数据点）竞品分析写到什么粒度？财务预测假设是怎么表述的？ |
| **语调** | 怎么在自信和审慎之间拿捏？风险部分怎么表达"我知道这个风险，我正在应对"而不显得在回避？创始人公开信用了什么人称和语气？ |
| **叙事** | 核心洞察是什么？怎么让每一章都服务于这个洞察？哪里用了故事，哪里用了数据？ |
| **格式** | 图表怎么编号和标注？引用什么格式？附录怎么组织的？ |

**Step 4.0.2: 输出对标分析摘要**

读完后，在对话中记录（不在计划书正文里）：

```
## 标杆学习摘要
阅读了：[列出实际读到的标杆名称和 URL]
核心发现：
- 结构方面：[1-2 句]
- 深度方面：[1-2 句，具体到你看到的最震撼的数据用法]
- 语调方面：[1-2 句]
应用到本次写作的具体做法：
- [3-5 条具体做法，例如"财务预测每张表都标注假设，像 Facebook S-1 那样"]
```

**Step 4.0.3: 对标检查（写完后执行）**

全部章节写完后，回头翻标杆学习摘要，逐条检查：
```
□ 结构对标：我的目录和标杆的目录，差距在哪？
□ 深度对标：标杆每千字 X 个数据点，我的是多少？差太多说明数据不够。
□ 语调对标：我的风险分析读起来像不像"我在回避"？对比标杆的风险分析——它们怎么写的？
□ 叙事对标：读完我的执行摘要，核心洞察能一句话说清吗？
```

**如果对标检查发现明显差距，修正后再进入 Step 5。**

### 4.1 写作规模与深度要求

创业计划书不是提纲，不是 PPT 讲义，而是一份完整、有深度、可以独立阅读的专业文档。老师或投资人拿到手，应该能从头到尾读懂项目的全部逻辑。

**写作铁律：**

1. **成文成段，严禁分点列举。** 所有内容必须以完整段落呈现。禁止出现"首先……其次……再次……"式的 bullet point 风格。每个观点都要展开论述，有分析、有数据、有逻辑推演。分点列举是提纲，不是计划书。

2. **规模参考。** 一份有说服力的创业计划书通常需要 15,000-25,000 字（中文）。对标 Alibaba S-1 "Business" 章的详尽程度、Rice Competition 获奖 BP 的逻辑密度、Google S-1 创始人公开信的坦诚语调。每章至少 3-5 个完整段落，核心章节（产品与服务、行业与市场分析、财务计划）应更详细。

3. **数据必须配分析，分析必须出结论，结论必须连回商业逻辑。** 不能只堆数据，要解释数据意味着什么机会或风险。不能只分析，要给出明确判断。每个判断都要说明如何影响定价、获客、成本或竞争优势。

4. **图表为辅，不能替代文字。** 图表用于直观展示数据，但每个图表前后必须有充分的文字分析和解读。不能出现"如下图所示"后只有图没有分析的情况。

5. **先写后改，一步到位。** 写的时候可以流畅写，不纠结措辞。全部章节写完后，用 Step 4.6 做去 AI 痕迹处理。

### 4.2 章节结构

根据 Step 0 加载的规范要求确定章节。标准 13 章模板如下：

1. 封面
2. 目录
3. 执行摘要
4. 企业介绍
5. 产品与服务
6. 行业与市场分析
7. 营销计划
8. 管理团队与组织
9. 生产运营
10. 财务计划
11. 风险管理
12. 退出策略
13. 附录

如规范文件有不同要求，以规范文件为准。

### 4.3 引用规范

**默认使用 GB/T 7714 顺序编码制**（上标序号¹²³）。若在 Step 0 检测到法大等高校格式规范文件，改用**脚注制**（每页重新编号，参考文献另起页分类著录），详见 `references/cupl-thesis-format.md`。

### 4.4 逐章撰写

每章写完向用户展示，收到确认后继续下一章。

**执行摘要撰写原则：**
- 最后写！所有章节完成后，提炼最精华的内容
- 1-2 页以内
- 必须包含：痛点、解决方案、市场机会、竞争优势、团队亮点、财务预测、资金需求

### 4.5 HTML 生成

**双层风格体系：学术结构 + 设计风格**

HTML 采用两层设计：
1. **学术骨架**（固定）：封面、目录锚点、页眉页脚、图表编号、打印友好
2. **设计皮肤**（来自 Step 1.5）：StyleKit 提取的颜色/字体/阴影/圆角 Token

**生成流程：**

1. 加载 Step 1.5 选定的 StyleKit 设计 Token
2. 将 Token 映射为 CSS 变量：
   ```css
   :root {
     --color-primary: /* StyleKit primary */;
     --color-secondary: /* StyleKit secondary */;
     --font-heading: /* StyleKit heading font */;
     --font-body: /* StyleKit body font */;
     --radius-base: /* StyleKit border radius */;
     --shadow-card: /* StyleKit shadow */;
     /* ... */
   }
   ```
3. 学术骨架使用 CSS 变量（而非硬编码颜色），确保换风格只需替换 Token
4. 封面页可适度使用设计风格（如 Logo + 品牌色），正文保持学术可读性
5. 图表配色使用 StyleKit Semantic Colors（成功/警告/信息色）

**必须包含：**
- 封面页（校名/公司名居中，logo/品牌色，时间。封面不遵循法大格式，采用设计风格）
- 自动生成的目录（带锚点链接）
- 页眉和页脚（如规范文件要求了页眉格式则严格遵循）
- 正文排版：遵循规范文件要求的字体字号行距；无规范时使用 StyleKit body font 或宋体，适当字号，1.5-1.75 倍行距
- 图表带编号和标题（图 1 xxx、表 1 xxx，居中编排）
- 引用标注：法大格式用脚注上标带圈数字，每页重新编号；默认用 GB/T 7714 上标序号¹²³
- 参考文献列表（另起页，按规范分类著录）
- 打印友好的 CSS（@media print — A4 纸张，正确页边距，避免表格/图片跨页断裂）
- 响应式设计（桌面 + 平板 + 移动端）

**法大格式专用规则：** 当检测到法大规范文件时，CSS 按 `references/cupl-thesis-format.md` 的完整映射生成（三级标题制、脚注、页眉页脚、A4 页面）。

**增强流程：**
1. 先检测 ui-ux-pro-max 是否可用，如可用则调用做品牌系统深化和视觉品质提升
2. 如果 StyleKit API 在 Step 1.5 时不可用但 ui-ux-pro-max 可用，用 ui-ux-pro-max 的设计系统能力作为降级方案
3. 两者都不可用时，使用纯学术报告风格（黑白灰 + 宋体）

HTML 文件保存为 `创业计划书.html`，放在项目根目录。

### 4.6 中文去 AI 痕迹化（humanizer-zh + 强化规则）

创业计划书作为课程作业或投资材料，AI 痕迹过重会被老师/投资人一眼识破。LLM 的中文输出有几个明显的"指纹"——破折号泛滥、双引号强调、公式化转折、段落结构千篇一律。本步骤在 HTML 正文撰写后、定稿前执行。**先写后改：写的时候可以流畅写，全部写完后再回头做去痕迹处理。**

**Step A: 调用 humanizer-zh**

检测 humanizer-zh 是否可用。如可用，用 Skill 工具调用 `humanizer-zh`，对以下关键章节逐章处理：
- 执行摘要
- 企业介绍
- 产品与服务描述
- 营销计划

处理目标：humanizer-zh quality score ≥ 40/50。

**Step B: 中文专项去痕迹规则（12 条）**

无论 humanizer-zh 是否可用，以下规则 MUST 逐条执行：

1. **破折号严格限制**：全文中连续两个破折号（——）不超过 5 处。这是 LLM 中文输出最明显的指纹——人类写中文很少频繁使用破折号。多余的破折号改为逗号、句号或分号。

2. **双引号限制**：除直接引用他人原话外，不使用双引号做强调。LLM 惯用的 "赋能"、"闭环"、"打法"、"私域" 等带引号的行业黑话，直接写成不加引号的自然表述。如需强调，用加粗或直接陈述。

3. **禁用 AI 标志词和句式**：检查并重写以下 LLM 高频模式——
   **过渡词和宣告句式：**
   - "值得注意的是" / "需要指出的是" / "应该看到" → 直接陈述，不预告
   - "综上所述" / "总而言之" / "总的来说" → 自然收尾，不宣告
   - "不可否认的是" / "毋庸置疑" / "毫无疑问" → 删除，直接说事实
   - "在这一背景下" / "从某种角度来说" / "从某种程度上" → 简化或删除
   **虚张声势的学术腔：**
   - "提供了有力支撑" → 用具体数据替代
   - "发挥着至关重要的作用" → 说明具体作用是什么
   - "具有重要的现实意义" → 说明对谁、在什么场景下有意义
   - "为进一步研究奠定了基础" → 如果是课程作业，这话轮不到你说
   **LLM 最爱的高频句式：**
   - "不仅……而且……" → 控制在全文 3 次以内
   - "从……到……再到……" → 全文最多 1 次
   - "一方面……另一方面……" → 全文最多 1 次
   - "首先……其次……最后……" → 全文最多 2 次（结构本身没问题，但 AI 用得太频繁了）
   - "可以说" / "应该说" / "必须承认" → 删除，这些是口语水词

4. **句式多样性**：连续 3 个句子不能以相同结构开头。检查 "通过……"、"随着……"、"在……下"、"根据……"、"由于……"、"为了……" 等介词开头，AI 最爱连续用。用短句打破节奏——偶尔用一个 5-8 字的干脆短句跟在长句后面，效果拔群。

5. **具体优于抽象**：用数字和事实替代模糊形容词。"市场很大" → "2025 年市场规模达 142.7 亿元"；"用户增长快" → "月活用户 3 个月内增长 230%"。如果一个陈述没有数字支撑，要么补数字，要么删掉。

6. **删除"水词"**：以下词除非有明确定义和必要，否则删除——
   "全方位"、"多层次"、"立体化"、"系统性"、"一站式"、"全链路"、"矩阵式"、"颗粒度"、"底层逻辑"、"顶层设计"、"差异化竞争"（没有差异化就没有竞争，这句话是废话）、"深耕"（99% 时候可以用"专注"替代）、"打造"（"建设"或"建立"更自然）、"赋能"（说清楚到底帮用户做了什么）

7. **专业术语自然化**：不要一上来就堆术语。先用人话解释概念，再在括号里给出术语。例如：先写 "平台从每笔交易中抽取一定比例的佣金"，再写 "(take rate)"。

8. **中文标点规范**：中文正文使用全角标点（，。！？）；引号使用中文引号（""）；数字与中文之间不加空格；英文单词前后加半角空格。

9. **段落结构多样化**：AI 写的段落几乎全是「总-分-总」结构——开头一句总结、中间展开、结尾再来一句总结。全文不能每段都是这个结构。部分段落可以用「问题-回答」结构（提一个问题，用整段回答），部分用「观点-反驳-修正」结构（先给常见看法、再说为什么不准、最后给出更准确的判断）。

10. **段落长度有起伏**：AI 倾向于每段 3-5 句话，像流水线出来的。人工检查时，确保有 1-2 句的短段落（比如单独一句强调一个关键数字），也有 8-10 句的长段落（需要展开复杂逻辑时）。一眼扫过去段落长短不一才对。

11. **语调分章节校准**：AI 的默认语调通篇一致，但不同章节应该有不同的语气——
    - 执行摘要：自信、简洁、有冲击力。每一句都在说服读者继续读。
    - 行业分析：冷静、客观、数据导向。像研究员在写报告，不像销售在推销。
    - 财务计划：保守、审慎、留有安全边际。不要用"乐观预计"这种词。
    - 风险分析：坦诚、不回避。最怕什么就写什么，然后写你怎么应对。

12. **朗读测试**：每改完一章，把它读出声来（或在脑子里默读）。如果你读到一个地方感觉"正常人不会这么说"——那就改。AI 写的文字有一个特征：看的时候觉得还行，读出来立刻觉得别扭。相信你的朗读直觉。

**Step C: 去痕迹质量检查**

随机抽取 HTML 中 5 段正文（每段 200 字以上，覆盖不同章节），用以下标准逐段评分：

| 检查项 | 标准 | 不通过措施 |
|--------|------|-----------|
| 破折号 | 全文 ≤ 5 处 | 逐处改为逗号或句号 |
| 句式多样性 | 无连续 3 句相同开头 | 打破句式，插入短句 |
| 标志词 | 无连续 5 个以上 Step B 所列词 | 逐词替换 |
| 段落结构 | 不是每段都「总-分-总」 | 改 2-3 段为其他结构 |
| 段落长度 | 有 <3 句的短段，也有 >6 句的长段 | 合并或拆分 |
| 朗读感 | 读出声来自然流畅，不机械 | 重写读起来别扭的句子 |
| 水词 | 无 Step B 第 6 条所列词（除非必要） | 删除或替换 |

不通过则重新修订对应段落，直到全部通过。

**Step D: 向用户展示**

将去痕迹后的关键段落（至少：执行摘要 + 2 段产品描述 + 1 段风险分析）展示给用户确认。询问："这些文字读起来自然吗？还有 AI 味吗？如果有，具体是哪句话让你觉得是 AI 写的？"

## Step 5: 质量检查

格式检查只是及格线。真正的好计划书要在深度、逻辑、说服力上经得起审视——读完应该让人觉得"这个团队真的懂行"，而不是"这个学生做了很漂亮的作业"。

### 5.0 答题板逐项核查

如果规范文件包含答题板或必答题清单，在常规检查前先逐项核对：

```
□ 答题板 1: [题目] → 已覆盖（第X章）/ 缺失 → 需补充
□ 答题板 2: [题目] → 已覆盖（第X章）/ 缺失 → 需补充
...
```

所有答题板必须 100% 覆盖。发现缺失立即补充到对应章节。

### 5.1 格式与完整性检查

对照 Step 0 的规范要求，逐项检查：

```
□ 所有要求章节是否都包含
□ 章节顺序和编号是否正确
□ 字数/页数是否达标（15,000-25,000 字）
□ 市场数据是否有真实来源 URL（每个数据点可追溯）
□ 学术引用是否按规范格式（GB/T 7714 或脚注制）
□ 参考文献是否完整（至少 5-8 篇）
□ 封面信息是否齐全
□ 目录是否正确锚定到各章节
□ HTML 打印效果是否正常（A4 纸，页边距正确）
□ 语法和错别字：全文至少通读一遍
□ 图表编号连续，标题居中
```

### 5.2 深度审计（逐章评分）

格式对了只是及格，内容有深度才是优秀。对每章做以下检查：

**执行摘要：**
```
□ 读完后能否用一句话复述核心论点？不能 → 重写
□ 是否包含：痛点、方案、市场、优势、财务、资金需求？缺一不可
□ 是否控制在 1-2 页？超了 → 删到只剩最关键的
```

**行业与市场分析：**
```
□ 数据后面有没有分析？只堆数字没有解读 → 补分析
□ 分析后面有没有结论？只分析没有判断 → 补判断
□ 结论有没有连回商业逻辑？"这个趋势意味着我们的定价策略应该是 X" 这样的句子每段至少一句
□ 竞品分析有没有深度？如果只写了竞品名字和一句话描述 → 回头做 Step 2 竞品分析
```

**财务计划：**
```
□ 收入预测有没有对标数据支撑？没有 → "基于 XX 公司 2024 年报数据推算" 这类标注必须出现
□ 有没有单位经济模型（至少：CAC、LTV、毛利率、LTV/CAC）？没有 → 这是真创业 BP 的灵魂，必须有
□ 成本结构有没有分类（固定/可变、一次性/经常性）？混在一起 → 拆开
□ 是否标注了预测假设和风险因素？没标 → 加了再交
```

**风险管理：**
```
□ 是否列出了最坏情况（不是"政策风险"这种泛泛的，是具体到"如果微信封了小程序入口怎么办"）？
□ 每个风险是否对应了具体的缓解措施（不是"加强管理"这种废话，是可操作的行动）？
□ 读起来是否在回避什么？如果读完感觉"这也太顺了" → 诚实度不够
```

### 5.3 逻辑流检查

从执行摘要到附录，检查全书的逻辑链条：
```
□ 执行摘要的核心论点 → 在行业分析里有数据支撑？
□ 行业分析的趋势判断 → 在营销计划里有对应的策略？
□ 营销计划的用户获取 → 在财务计划里有对应的获客成本？
□ 财务计划的收入预测 → 在产品描述里有对应的收费逻辑？
□ 风险分析列出的最大风险 → 在其他章节有应对措施的呼应？
```

如果某章的结论在后续章节毫无体现——那章就是白写的。要么删，要么连线。

### 5.4 对比审查：真创业 BP 标准

用 Step 0.2 的真创业 BP 对标标准做最后一轮审查：

```
□ 这份计划书的核心洞察是什么？一句话能说清吗？
    Airbnb 的洞察：旅行者愿意住陌生人家，只要价格够低、体验够真
    Uber 的洞察：每个人口袋里都有 GPS，按个按钮就能叫车
    你的洞察：______（填不出来就回去想，这是真创业 BP 的灵魂）

□ 读完会不会觉得"这个团队真的懂行"？
    还是觉得"这个学生做了很漂亮的作业"？
    区别在于细节：有没有具体的运营数据、真实的用户反馈、可操作的执行计划

□ 有没有在回避什么？
    投资人/老师最怕的不是风险——是没有提到他们担心的风险。
    主动把你最怕被问到的问题写进风险分析里，然后写你怎么应对。

□ 每段话都在为你的核心论点添砖加瓦吗？
    不是 → 删。不管写得多好，分散注意力的内容就是噪音。
```

### 5.5 检查方法

1. 重新阅读规范文件，逐条对照
2. 抽查 3-5 个数据引用，确认 URL 可访问
3. 用浏览器打开 HTML，检查排版和打印预览
4. 把执行摘要发给一个不了解项目的朋友，问他读懂了没有、核心论点是什么——答不出来说明执行摘要写砸了

### 5.6 评分量规

按以下维度自评（每项 1-10 分）：

| 维度 | 10 分标准 | 及格线 |
|------|----------|--------|
| 核心论点清晰度 | 一句话能说清，所有章节都服务于这个论点 | 7 |
| 数据质量 | 每条数据有来源 URL，有分析有结论，财务有对标 | 8 |
| 分析深度 | 不堆数据，每个数据解释了"然后呢？" | 6 |
| 竞品理解 | 对核心竞品的融资/产品/模式有深度了解 | 7 |
| 财务可信度 | 有单位经济模型，有对标参照，假设标注清楚 | 7 |
| 诚实度 | 不回避硬问题，风险分析具体而非泛泛 | 7 |
| 文字自然度 | 通过 Step 4.6 全部检查，读出声自然流畅 | 7 |
| 格式规范 | 完全符合规范文件或标准模板要求 | 9 |

总分 < 55 → 不提交，回头改。总分 55-65 → 可以交，但还有提升空间。总分 > 65 → 这是一份有说服力的计划书。

### 5.7 完成报告

全部检查通过后，输出完成报告：

```
## 创业计划书撰写完成

**文件位置：** 创业计划书.html
**总字数：** XXXX 字
**章节数：** X 章
**数据引用：** X 处（含 URL）
**学术文献：** X 篇（GB/T 7714 格式）
**深度评分：** XX/80（6.6 评分量规）
**使用的增强技能：** pm-skills[X项], gstack[X项], superpowers[X项], mattpocock[X项], ui-ux-pro-max[X项]

**下一步建议：**
1. 用浏览器打开 HTML，检查排版
2. 打印预览确认分页效果
3. 如需 Word 版，可复制 HTML 内容到 Word
4. 准备答辩陈词（使用 Step 6 路演模式）
```

## Step 6: 路演与答辩准备

商业路演和学术答辩的根本区别：路演的听众手里有钱（或分数），你的每一秒都在争取他们继续听下去的兴趣。讲完不能被记住的路演等于没讲。

### 6.1 路演 PPT 大纲

按商业路演标准生成 10-12 页 PPT 大纲。路演不是念计划书——每页只说一件事，视觉优先，文字极少。

**标准路演结构（12 页）：**

| 页码 | 内容 | 要点 |
|------|------|------|
| 1 | 封面 | 项目名称 + 一句话价值主张 |
| 2 | 痛点 | 一个具体场景，让听众立刻感同身受 |
| 3 | 解决方案 | 你的产品如何直击这个痛点，一句话说清 |
| 4 | 产品 Demo/截图 | 核心功能展示，图片为主，不要文字说明 |
| 5 | 市场机会 | TAM/SAM/SOM 数据，用可视化图表，不要表格 |
| 6 | 商业模式 | 怎么赚钱，收入结构和定价逻辑，一页画清楚 |
| 7 | 竞争优势 | 与竞品的差异化对比，一个维度的差异 > 十个维度的对比 |
| 8 | 增长策略 | 获客渠道、用户增长路径、关键指标 |
| 9 | 技术/运营壁垒 | 护城河是什么——专利/数据/网络效应/品牌/供应链 |
| 10 | 团队介绍 | 核心成员 + 为什么你们是唯一能做这件事的人 |
| 11 | 财务预测 | 3 年收入/成本/利润，融资需求和使用计划 |
| 12 | 结束页 | 联系方式 + 品牌 Slogan，不要写"感谢聆听" |

**PPT 设计原则（商业路演标准）：**
- 每页不超过 3 个要点，超过则分页
- 字号不小于 24pt（32pt 更安全——投屏到投影仪会缩小）
- 图表优于文字，高清图片优于图表——人脑处理图片比文字快 6 万倍
- 优先使用 StyleKit 品牌色（Step 1.5 选定的 Token），整份 PPT 只用 3 种颜色
- 绝对不要出现大段文字——听众不会读，只会觉得你偷懒
- 动画不要花哨——淡入淡出足够，弹入弹出很廉价
- 每页停留 30-60 秒，整场路演控制在 5-8 分钟

**多格式适配：**

| 格式 | 时长 | 页数 | 重点 |
|------|------|------|------|
| 电梯路演 | 60-90 秒 | 无需 PPT | 只有痛点→方案→为什么你能做→你缺什么 |
| 投资人路演 | 10 分钟 | 12 页全用 | 加上 Demo + 深度财务 + 退出策略 |
| 课程答辩 | 5-8 分钟 | 10-12 页 | 老师更关注逻辑完整性，多讲市场分析 |
| 比赛路演 | 5 分钟 | 8-10 页 | 评委看创新性和可行性，痛点要刺眼 |

### 6.2 答辩陈词

生成路演逐字稿（约 1000-1200 字），格式为 "讲什么 + 翻到第几页"。不是念稿，是引导听众注意力的路线图。开头 30 秒定生死——不要寒暄，不要自我介绍（封面上有），直接进入最痛的那个画面。

示例结构：
```
[P1 封面]
各位老师好，今天带来的项目是[项目名称]——[一句话价值主张]。

[P2 痛点]
想象一下这个场景：[具体的、能让听众皱眉头或点头的场景]...
这个问题每年困扰着 [X 万] 人，他们目前的解决方案是 [竞品/替代方案]，
但这行不通因为 [竞品最大的缺陷]。

[P3 解决方案]
所以我们做了 [产品名称]。一句话：[核心功能的一句话描述]。
```

**逐字稿写作要点：**
- 每页 PPT 对应 30-60 秒的口头讲解（约 80-150 字中文）
- 用口头语言，不要用书面语——"咱们来看" 而不是 "我们将展示"
- 数字要说人话——"140 亿" 而不是 "14,200,000,000 元"
- 模拟讲的时候发现卡顿的地方 → 修改 PPT 或修改稿子
- 结尾要有行动指引——"如果你认同这个问题值得被解决，请…"

### 6.3 常见问题预演

根据计划书内容预判 15-20 个最可能被问的问题，准备简洁回复。问题越刁钻，准备越要提前。

**投资人视角的问题（如果你在融资）：**

| 类别 | 典型问题 |
|------|---------|
| 市场 | "这个市场真有你说的那么大？""如果巨头进来做怎么办？""为什么是现在？" |
| 产品 | "用户为什么非用你不可？""竞品抄你要多久？""你的核心功能真的比 XX 好用吗？" |
| 商业模式 | "单位经济模型什么时候打平？""获客成本多少？怎么降？""LTV/CAC 多少？" |
| 技术 | "为什么不直接用现成的 SaaS？""你的 AI 模型有什么数据优势？""安全怎么做？" |
| 团队 | "你为什么是唯一能做这件事的人？""你还缺什么人？""股权怎么分的？" |
| 财务 | "为什么第三年能翻 10 倍？""你打算融多少？钱花在哪？""退出路径是什么？" |
| 陷阱题 | "你觉得你最大的劣势是什么？""如果今天就要倒闭了，最可能的原因？""你上一轮投资人怎么说你的？" |

**老师/评委视角的问题（如果是课程作业）：**

| 类别 | 典型问题 |
|------|---------|
| 理论 | "你的商业模式和 XX 理论有什么关系？""你参考了哪些创业案例？" |
| 逻辑 | "你这个预测的假设是什么？""数据来源可靠吗？""竞争分析是不是太乐观了？" |
| 深度 | "你有没有自己做过用户调研？""有没有考虑过政策风险？""可持续竞争优势是什么？" |

每个问题和回复控制在 150-200 字。回复结构：一句话结论 + 一个数据/事实支撑 + 一个"所以"。不要绕弯子——评委问了就直接答，答完就停。

### 6.4 路演后材料

路演结束后应该留给听众的东西：

1. **执行摘要（1 页 PDF）**：比 PPT 更详细一点的文字版，方便老师/投资人回去翻
2. **联系方式**：微信二维码 + 邮箱，放在执行摘要页脚
3. **下一步**：如果老师/投资人感兴趣，你希望他们做什么——约时间深聊？试用产品？介绍客户？

## 交互原则

1. **每步确认，不跳步。** 但也不要事无巨细征询——数据搜索和分析可以在一个步骤内批量完成
2. **数据不能编造。** 找不到数据就说"未找到官方数据，使用估算"，并标注
3. **中文撰写，专业但不学术腔。** 让投资人和老师都能看懂
4. **始终保持用户知情权。** 搜索了什么关键词、找到了什么数据源，都要汇报

各步骤的内联指令已包含依赖可用/不可用的处理逻辑。**原则：有哪个用哪个，不等待，不因依赖缺失中断工作流。** 详细降级策略见各步骤内联说明。
