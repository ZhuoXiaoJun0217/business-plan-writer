# Step 3: 三引擎五阶段详细流程

> 此文件是 Step 3 技术方案的详细执行手册。当 Step 3.1 判断项目走路径 A（完整技术方案）且至少一个引擎可用时，按此流程执行。

---

## 阶段一：技术规划（superpowers 主导，mattpocock 压测）

1. 调用 `superpowers:writing-plans`：撰写技术方案规划，明确目标、范围、排除项
2. 调用 `superpowers:brainstorming`：与技术方案相关的设计决策头脑风暴
3. 调用 `grill-me`（mattpocock）：对技术方案进行压力测试，逐一推敲 design tree 的每个分支，确保方案经得起追问
4. 产出：技术选型矩阵（含决策理由）、系统边界定义、关键风险点
5. **产出 ADR（架构决策记录，≥3 条）**，每条格式：
   ```
   ADR-001: [决策标题]
   背景：[什么情况下需要做这个决策]
   决策：[选了什么方案，为什么]
   后果：[这个决策会导致什么——好的和坏的都写]
   替代方案：[考虑过但没选的方案，为什么没选]
   ```
6. **产出规划移交摘要**（3-5 行），包含：关键决策、产出位置、留给下阶段的风险

引擎不可用时降级：superpowers 不可用 → 自建规划模板；mattpocock 不可用 → 内置"魔鬼代言人"问答（5 轮追问）。

---

## 阶段二：架构设计（gstack 主导，mattpocock 增强）

1. 调用 gstack `/plan-eng-review`：系统架构评审，锁定数据流、边缘情况、测试策略
2. 调用 gstack `/design-consultation`：产品设计系统构建（如需要 UI）
3. 调用 `grill-with-docs`（mattpocock）：用领域模型挑战设计方案，打磨术语精确性，产出 ADR
4. 调用 `improve-codebase-architecture`（mattpocock）：识别架构深化机会，确保可测试性
5. **产出安全自查清单**（见下方安全门禁节），标注每个安全项的适用性（适用/不适用/待定）
6. 产出：架构图（Mermaid）、API 设计概要、数据模型、组件树
7. **产出架构移交摘要**（3-5 行），包含：架构关键决策、安全项标注、留给实现阶段的风险

引擎不可用时降级：gstack 不可用 → 内置架构评审清单（10 项）；mattpocock 不可用 → 内置领域模型问答。

---

## 阶段三a：原型验证（mattpocock 主导，可丢弃原型）

**此阶段目标仅验证技术可行性，不追求代码质量、不写测试、不考虑生产环境。**

1. 调用 `prototype`（mattpocock）：构建可丢弃原型，快速验证：
   - 数据模型是否成立（核心实体关系是否能跑通）
   - 状态管理方案是否合理（关键状态流转是否能工作）
   - 核心技术风险是否可解（最担心的那块能不能做出来）
2. 如果 `prototype` 不可用：自建单文件 HTML/JS 原型，只做核心流程的 mock
3. **不在此阶段写测试**——原型验证通过后可以扔掉代码，测试在三b 阶段从零开始写
4. 产出：可运行原型（标注"可丢弃，仅验证可行性"）
5. **产出原型验证记录**（3-5 行）：哪些假设被验证了、哪些推翻了、哪些还不知道

引擎不可用时降级：mattpocock 不可用 → 自建 HTML/JS 单文件原型。

---

## 🔒 安全门禁（阶段三a 之后，三b 之前）

**触发场景判断：** 检查项目是否涉及以下任一场景。涉及则此门禁强制执行，不通过不进入三b：

| 场景 | 检查项 |
|------|--------|
| 用户注册/登录/身份认证 | 密码哈希方案、会话管理、MFA 支持 |
| 支付/交易/资金处理 | 支付网关选择、PCI-DSS 合规、资金流向 |
| 用户个人数据收集与存储 | 加密存储、数据最小化、隐私政策 |
| 第三方 API 集成 | API Key 管理、依赖安全审计、降级策略 |
| 用户生成内容（UGC） | 内容审核、敏感词过滤、举报机制 |

**安全清单（逐项打勾）：**

```
□ 认证安全
  □ 密码使用 bcrypt/argon2 哈希存储（非明文、非 MD5、非 SHA1）
  □ 会话 Token 设置 httpOnly + Secure + SameSite
  □ 是否支持 MFA（多因素认证）？如是创业项目，至少计划在 V1.0 加入

□ 数据安全
  □ 敏感数据（密码、支付信息、个人身份信息）加密存储
  □ 全站 HTTPS，HSTS 头设置
  □ 数据库有自动备份策略和灾难恢复计划

□ 隐私合规
  □ 是否收集了不必要的用户数据？（最小化原则）
  □ 是否有隐私政策页面？（至少说明收集什么、怎么用、怎么删）
  □ 是否符合《个人信息保护法》？（中国用户数据本地化存储、用户同意、删除权）

□ API 安全
  □ 关键接口有 rate limiting（防暴力破解/爬虫）
  □ 有 CSRF 保护（非 GET 请求需 Token 验证）
  □ 输入校验在服务端执行（前端校验只是 UX，不是安全）

□ 依赖安全
  □ 第三方 SDK/API Key 不硬编码在代码中（使用环境变量）
  □ 关键依赖固定版本号，定期检查已知漏洞
  □ 第三方服务有降级方案（支付挂了、地图挂了怎么处理）

□ 支付安全（如涉及支付）
  □ 使用成熟的支付网关（微信支付/支付宝/Stripe）而非自建支付逻辑
  □ 不在服务端存储完整信用卡号
  □ 退款/争议流程有明确的产品设计
```

**门禁结果记录：**
```
安全门禁：通过 / 有条件通过 / 不通过
高危项：[列出，如有则必须修复才能进入三b]
中低危项：[列出，记录为已知风险，在三b 中处理]
不适用项：[列出，说明原因]
```

**无安全场景的项目（纯静态展示、本地工具等）跳过此门禁**，直接进入三b。

## 阶段三b：正式实现（mattpocock 主导，superpowers 配合）

**此阶段从零开始写生产级代码。原型阶段的代码不继承，全部重写。**

1. 调用 `tdd`（mattpocock）：以 red-green-refactor 循环做测试驱动开发
2. 调用 `superpowers:test-driven-development`：补齐集成测试，确保测试覆盖率
3. 调用 gstack `/design-html`：生产级 HTML/CSS 实现
4. 产出：测试套件（单元测试 + 集成测试）、生产级代码
5. **产出实现移交摘要**（3-5 行）：测试覆盖率、已知技术债务、留给评审阶段关注点

引擎不可用时降级：mattpocock 不可用 → 手动 red-green-refactor + 自建原型演进为生产代码；superpowers 不可用 → 手动测试 checklist。

---

## 阶段四：评审验证（三引擎合流）

1. 调用 gstack `/review`：代码审查，检查正确性
2. 调用 `diagnose`（mattpocock）：对审查发现的问题做纪律化诊断（Reproduce → Minimise → Hypothesise → Instrument → Fix）
3. 调用 `superpowers:verification-before-completion`：对照 PRD 需求逐条验证
4. 如产品可部署，调用 gstack `/qa`：真实浏览器测试
5. **成本回溯验证**：对比阶段 3.2.2 的成本估算与阶段三b 实际的资源消耗
   ```
   □ 云服务实际选型是否在预算内？（对比估算值）
   □ 第三方 API 调用量预估是否仍然成立？
   □ 开发周期偏差（实际 vs 计划，差了多少天）
   □ 如有重大偏差（>30%），是否需要调整方案或告知用户？
   ```
6. 产出：评审报告、问题清单、修复记录、成本回溯结论
7. **产出最终评审报告**（5-8 行）：整体评估、遗留风险、部署建议、成本符合度

引擎不可用时降级：gstack 不可用 → 手动代码审查 + 手动浏览器测试；mattpocock 不可用 → 内置诊断流程；superpowers 不可用 → 自建 checklist 逐项打勾。

---

## 引擎不可用时的简化

- 某个引擎不可用 → 按每阶段的降级策略处理，用剩余引擎完成
- 某引擎的部分技能不可用 → 使用 SKILL.md 3.5 节的技能级降级替代
- 全部不可用 → 使用 SKILL.md 3.7 节的全内置后备流程

核心原则：技能级降级优先于引擎级降级。有哪个用哪个，不等待。永远不因依赖缺失而中断工作流。
